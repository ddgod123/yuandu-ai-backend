# 端到端流程细化

## 1. 推荐的生产流程

在现有实现上，推荐把 GIF 流程固化成下面这条标准链：

```text
CreateVideoJob
  -> Admission 校验
  -> archive/public 任务写入
  -> Asynq 入队
  -> Processor 抢占执行
  -> 下载源视频
  -> ffprobe / auto crop
  -> local candidate recall
  -> AI1 Director
  -> AI2 Planner
  -> feedback rerank / diversity / threshold
  -> candidate persist
  -> render windows resolve
  -> GIF render + loop tune
  -> output evaluate
  -> AI3 Judge
  -> hard gate
  -> deliver/internal/reject/manual_review
  -> upload / collection / emoji / zip / public sync
  -> cost sync / points settlement
  -> cleanup
```

## 2. 状态机细化

### 建议使用原则

- 主 stage 面向用户/平台整体进度
- 子阶段面向 GIF AI 内部审计

---

## 2.2 GIF 子阶段

当前代码已有：

- `briefing`
- `planning`
- `scoring`
- `reviewing`

### 建议语义

| 子阶段 | 含义 | 当前代码落点 |
|---|---|---|
| `briefing` | AI1 生成任务 brief | `requestAIGIFPromptDirective` |
| `planning` | AI2 在候选上生成 proposal | `requestAIGIFPlannerSuggestion` |
| `scoring` | 候选落库、反馈 rerank、渲染窗口确定 | `processor_pipeline.go` |
| `reviewing` | AI3 复审成品输出 | `runAIGIFJudgeReview` |

### 建议再补一条内部约束

只有 `containsString(generatedFormats, "gif") == true` 时，`reviewing` 才允许进入 `running`。

当前代码已经接近这个逻辑，建议继续保持。

## 3. Admission 阶段细化

## 3.1 输入

- 用户信息
- 标题/分类
- `source_video_key`
- output formats
- edit options
- priority

## 3.2 核心动作

1. 标准化 source key
2. 校验输出格式只能走单格式任务
3. 检查 ffmpeg/ffprobe
4. probe 源视频
5. 推导建议质量模板
6. 估算积分与成本
7. 写 archive/public job
8. 冻结积分
9. 投递队列

## 3.3 失败策略

- 参数错误：同步 400
- 资源缺失：同步 400/409
- 队列不可用：同步 500
- 积分不足：同步业务错误

### 这里不建议做的事

- Admission 阶段调用 AI
- Admission 阶段下载全量视频并做重分析

---

## 4. Candidate Recall 阶段细化

## 4.1 输入

- `probeVideo` 元数据
- auto crop 结果
- quality settings
- 原始视频路径

## 4.2 输出

- `highlightSuggestion`
- `highlightCandidate[]`
- 每个 candidate 的：
  - 时间窗
  - base score
  - reason
  - confidence
  - scene score

## 4.3 细化建议

建议把候选召回进一步拆成三步：

### 第一步：粗召回

优先找出：

- 节奏变化点
- motion 高点
- scene 切换点
- 画面质量可接受段

### 第二步：候选清洗

- 去非法窗口
- 去极短窗口
- 去高度重叠窗口
- 去明显模糊/黑屏/曝光异常窗口

### 第三步：候选多样性整理

- 时间段分散
- 场景分散
- 语义 reason 分散
- 避免多个候选扎堆在同一情绪峰值附近

---

## 5. AI1 Director 阶段细化

## 5.1 输入建议

Director 的输入应以“任务摘要”为主，而不是全量候选细节：

- 视频 meta
- 时长/FPS/尺寸
- 任务格式
- 输出上限
- 用户/场景信息（如果可用）
- local recall 摘要
- 运营层 instruction

## 5.2 输出建议

继续沿用当前结构化字段，但要求在 metadata 里再保留：

- `format`
- `stage=ai1`
- `operator_enabled`
- `operator_version`
- `fixed_prompt_version`
- `fallback_reason`
- `experiment_bucket`（建议新增）

## 5.3 失败处理

### 推荐回退顺序

1. 优先走可用 prompt template
2. template 不可用则退 built-in prompt
3. AI 调用失败则写 fallback directive
4. 主链路继续

### 结果要求

无论成功失败，都应尽量在 `archive.video_job_gif_ai_directives` 有落点。

---

## 6. AI2 Planner 阶段细化

## 6.1 输入建议

Planner 的输入要尽量结构化，减少 prompt 漂移：

- director brief
- topK candidates
- 每个 candidate 的：
  - `start_sec/end_sec`
  - `score`
  - `reason`
  - `scene_score`
  - `candidate_id`（建议进入上下文）
  - 未来可扩展的：clarity/motion/loop_estimate
- 视频总时长
- 目标输出数

## 6.2 输出建议

当前 proposal 结构已经足够好，建议继续保留：

- `proposal_rank`
- `start_sec/end_sec`
- `score`
- `proposal_reason`
- `semantic_tags`
- `expected_value_level`
- `standalone_confidence`
- `loop_friendliness_hint`

并建议在 `metadata` 再写入：

- `source_candidate_ids`
- `director_brief_version`
- `selection_policy_version`
- `diversity_bucket`
- `experiment_bucket`

## 6.3 后处理建议

Planner 返回后，不直接用其排序结果，而是继续经过：

1. 时间窗 clamp
2. proposal rank 去重
3. 非法 proposal 过滤
4. 非重叠筛选
5. confidence threshold 过滤
6. topN 截断
7. 写 proposal -> candidate 映射

当前代码已实现其中大部分，建议保持“AI 提名 + 系统约束”的模式。

## 7. Feedback Rerank 阶段细化

当前代码已经有：

- 用户历史偏好
- public feedback 信号
- negative guard
- rerank logs

### 建议把 rerank 放在固定位置

```text
AI2 Planner 之后，render windows resolve 之前
```

### 理由

- 候选还没变成实际 output，调整成本最低
- 可把用户长期偏好融入窗口选择
- 不会污染技术评测事实层

### 建议补的审计字段

在 `ops.video_job_gif_rerank_logs.metadata` 中统一记录：

- `profile_version`
- `weighted_signals`
- `reason_preference`
- `negative_guard_hit`
- `selected_before`
- `selected_after`
- `experiment_bucket`

---

## 8. Scoring / Render Selection 阶段细化

当前渲染前已存在：

- 候选输出上限
- 置信度阈值
- size budget
- 长视频降配策略

### 推荐逻辑顺序

```text
candidate/proposal pool
  -> confidence gate
  -> overlap gate
  -> diversity gate
  -> size budget gate
  -> output limit gate
  -> animated windows
```

### 为什么 size budget gate 要在 output limit 前

因为很多情况下不是“候选太多”，而是“候选组合后总体体积超预算”。

## 9. Render 阶段细化

## 9.1 输入

- animated windows
- source path
- quality settings
- GIF 目标 size/fps/max_colors
- loop tune 开关

## 9.2 输出

- `public.video_image_outputs`
- `archive.video_job_gif_evaluations`
- collection / emoji / artifact

## 9.3 这里的关键原则

Render 只产出事实：

- 文件多大
- 分辨率如何
- 时长多少
- loop 是否改善
- 技术分如何

不要在 render 层做“这个 GIF 值不值得传播”的主观判断。

## 10. AI3 Judge 阶段细化

## 10.1 输入建议

Judge 输入应包含四类上下文：

1. **成品事实**
   - output_id
   - size/width/height/duration
   - output score
2. **技术评测**
   - overall/emotion/clarity/motion/loop/efficiency
3. **上游上下文**
   - director brief 摘要
   - planner proposal reason
   - semantic tags
4. **业务上下文**
   - requested_format
   - deliver count target
   - 场景标签（如果可用）

## 10.2 输出建议

保留当前四类 recommendation：

- `deliver`
- `keep_internal`
- `reject`
- `need_manual_review`

同时在 metadata 里建议明确：

- `hard_gate_applied`
- `hard_gate_reason_codes`
- `judge_prompt_version`
- `director_prompt_version`
- `planner_prompt_version`
- `decision_policy_version`

## 11. Hard Gate 阶段细化

当前代码已经具备以下硬门槛思路：

- overall score 下限
- clarity 下限
- loop 下限
- output score 下限
- duration 下限
- severe oversize 阈值
- evaluation 缺失保护

### 推荐固化为两级 gate

#### Gate A：绝对阻断

命中即 `reject`：

- output 丢失
- evaluation 丢失且无人工兜底策略
- duration 明显非法
- size 严重超预算
- 清晰度极低/损坏

#### Gate B：人工复审触发

命中即 `need_manual_review`：

- 边界分数
- judge 与 technical facts 强冲突
- 语义价值高但技术分略低
- 运营任务特殊场景

---

## 12. 交付决策建议

推荐把最终决策写成一段非常明确的策略：

```text
if GateA fail:
  reject
else if Judge says manual_review:
  need_manual_review
else if GateB hit:
  need_manual_review
else if Judge says reject:
  reject
else if Judge says keep_internal:
  keep_internal
else:
  deliver
```

### 这样做的好处

- 冲突顺序固定
- 容易复盘
- 容易写测试
- 容易给后台解释

---

## 13. Rerender 流程细化

当前已有：`POST /api/admin/video-jobs/:id/rerender-gif`

### 推荐把它定义成“轻量补偿链路”

```text
load job
  -> load proposal
  -> load collection
  -> download source
  -> clamp proposal window
  -> render one GIF
  -> write output / artifact / emoji
  -> invalidate zip if needed
  -> write cost delta
  -> optional default review = need_manual_review
```

### 最佳实践

重渲默认：

- 不重新跑 Director
- 不重新跑 Planner
- 不重新跑 full candidate recall
- 只复用 proposal 做窗口渲染

这是成本最优方案。

---

## 14. 失败回退链路细化

## 14.1 AI 失败回退

- Director 失败 -> fallback directive
- Planner 失败 -> local candidates topN
- Judge 失败 -> 保守策略：`keep_internal` 或 `manual_review`

## 14.2 渲染失败回退

- timeout -> 短时长重试
- size 超限 -> 降低 fps/max_colors
- loop tune 失败 -> 回退 base window

## 14.3 存储失败回退

- 上传失败 -> 保留局部结果与事件日志
- public sync 失败 -> 允许补偿任务
- zip 失败 -> 不影响主 output 完成状态

---

## 15. 端到端验收标准

一条健康的 GIF 任务，在后台应至少能回答：

1. 为什么候选是这些？
2. Director 给了什么 brief？
3. Planner 为什么选这些 proposal？
4. 为什么最后选中了这些 output？
5. AI Judge 为什么这么建议？
6. hard gate 有没有降级？
7. 用户反馈后排序为什么变了？
8. 如果重渲，成本和结果差异是什么？

如果这 8 个问题都能在系统里回答清楚，这条链路就是合格的生产链。
