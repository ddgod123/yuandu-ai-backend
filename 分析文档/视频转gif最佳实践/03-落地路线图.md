# 视频转 GIF 最佳实践落地路线图

> 目标：在不打断现有生产链路的前提下，逐步把当前系统升级成“更稳、更准、更省”的 AI GIF 平台。

## 1. P0：先把衡量体系补全（1~2 周）

这一步不要急着改模型，先把“什么叫变好”定义清楚。

### 1.1 建议先补齐的线上指标

- 任务成功率
- AI Director / Planner / Judge 调用成功率
- fallback 率
- proposal 接受率
- Judge 降级率
- hard gate 拦截率
- deliver / keep_internal / reject / manual_review 分布
- 单任务 AI 成本
- 单个 deliver GIF 的平均成本
- rerender 率
- manual review 率
- 用户反馈正负样本比

### 1.2 建议先补齐的审计链路

确保一个任务能串起：

```text
job -> candidate -> proposal -> output -> evaluation -> review -> feedback -> rerender
```

### 1.3 P0 完成标准

- 可以按 prompt version 看效果
- 可以按视频类型看 deliver 率
- 可以按用户群看成本差异
- 可以快速定位 AI 失败、技术失败、业务失败分别发生在哪层

---

## 2. P1：优化候选召回与 Planner 输入（2~4 周）

这是最容易提升命中率的一步。

### 2.1 优先做的事

#### A. 候选多样性策略

在现有非重叠基础上增加：

- 场景多样性
- 时间段多样性
- 语义标签多样性
- 镜头节奏多样性

#### B. Planner 输入摘要增强

给每个 candidate 增加更结构化的摘要特征，例如：

- 关键帧摘要
- 首尾相似度
- motion 统计
- clarity / loop 预估
- 可选的 OCR / face / subtitle 摘要

#### C. 统一 Director 权重消费

确保 Planner 在 ranking 时真正参考：

- `quality_weights`
- `loop_preference`
- `duration_pref_*`
- `must_capture / avoid`

### 2.2 P1 完成标准

- proposal top1 命中率提升
- Judge 打回率下降
- rerender 率下降
- 候选重复/扎堆现象明显减少

---

## 3. P2：把 Judge 与 hard gate 校准成稳定的交付门禁（2~4 周）

### 3.1 优先做的事

#### A. 固化 Judge 规则

明确文档与代码规则：

- Judge 只能做语义层判断
- hard gate 永远高于 Judge
- evaluation 缺失时默认保守处理

#### B. 建立边界样本池

重点抽样这些 case：

- Judge deliver 但 hard gate reject
- Planner 高分但 Judge reject
- 用户正反馈但 Judge keep_internal
- 用户负反馈但系统 deliver

#### C. 加入人工校准回路

让运营或审核同学定期复核边界样本，并回写：

- baseline
- manual scores
- prompt 调整建议
- hard gate 阈值调整建议

### 3.2 P2 完成标准

- Judge 与 hard gate 冲突率下降
- manual review 样本更集中在真正模糊 case
- deliver 的稳定性更高

---

## 4. P3：把成本做成策略化控制（2~3 周）

### 4.1 推荐上线的策略

#### A. 轻链路模式

适用于：

- 短视频
- 本地高置信候选
- 普通用户任务

链路示例：

```text
local recall -> render/evaluate -> optional judge
```

#### B. 标准链路模式

适用于多数常规任务：

```text
local recall -> AI2 Planner -> render/evaluate -> AI3 Judge -> hard gate
```

#### C. 高质量链路模式

适用于：

- 高价值运营任务
- 高付费用户
- 复杂内容视频

链路示例：

```text
AI1 Director -> AI2 Planner -> rerank -> render/evaluate -> AI3 Judge -> manual review(optional)
```

### 4.2 P3 完成标准

- 单任务平均 AI 成本下降
- 成本下降时 deliver 率无明显恶化
- 高价值任务仍能保持高质量链路

---

## 5. P4：把人工审核与重渲做成运营工具（2~4 周）

### 5.1 推荐建设内容

- proposal 列表页：可按 rank / score / reason / tag 查看
- output 审计页：可串看 evaluation / review / feedback
- 一键按 proposal 重渲
- 重渲时支持不同渲染策略模板
- 支持对历史任务做小规模回放实验

### 5.2 为什么值得做

因为线上最真实的调优往往不是“改 prompt 再看平均分”，而是：

- 运营挑具体 case
- 对比不同窗口/不同渲染参数
- 看实际交付效果

而你已经有 `gif_rerender.go` 的基础，非常适合继续做成正式工具链。

---

## 6. P5：建立长期回归与数据闭环（持续建设）

### 6.1 建议沉淀的数据资产

- 高质量交付样本集
- Judge 争议样本集
- 用户强正反馈样本集
- 用户强负反馈样本集
- manual review 标注集
- prompt version 对照集

### 6.2 建议固定节奏

- 每周：看 baseline 与异常波动
- 每两周：复盘 prompt 调整效果
- 每月：回看 deliver 业务效果与成本
- 每季度：决定是否引入新 CV 能力或更强多模态模型

### 6.3 长期方向

等你把上述闭环跑顺后，再考虑进一步增强：

- 更强的候选特征提取
- 垂直小模型辅助召回
- 更细的用户偏好建模
- 面向视频类型的分域策略

不要一开始就追求“大模型端到端替代整个系统”。

---

## 7. 优先级建议（最重要）

如果只按投入产出比排序，我建议是：

1. **先补指标和审计链路**
2. **再补候选召回多样性与 Planner 输入质量**
3. **再做 Judge / hard gate 校准**
4. **再做成本分层策略**
5. **最后做更重的 AI 能力增强**

原因很简单：

- 没有指标，无法证明优化有效
- 候选质量差，后面 AI 再强都救不回来
- Judge 不稳定，会导致交付门禁抖动
- 成本不分层，链路越做越贵

---

## 8. 最终落地判断标准

当你把这套最佳实践真正落地后，应该能看到这些结果：

- AI 失败不会拖垮主链路
- 不同 prompt 改动可以被量化评估
- deliver 结果更稳定
- rerender 从救火行为变成运营优化手段
- manual review 从成本项变成训练数据入口
- 单个优质 GIF 的综合成本持续下降

这才说明你的系统真正从“能跑”进化到了“能持续优化”。
