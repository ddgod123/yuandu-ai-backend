# 总体架构与职责边界

## 1. 目标架构

基于当前仓库，建议把视频转 GIF 系统明确分成 8 层：

```text
[1] API / Task Admission
    -> 参数校验 / 资源校验 / 成本预估 / 积分冻结
[2] Media Preprocess
    -> probe / crop / 基础时长与画面分析
[3] Candidate Recall
    -> 本地候选召回 / 去重 / 多样性保留
[4] AI Decision Layer
    -> AI1 Director / AI2 Planner / AI3 Judge
[5] Render & Evaluate
    -> GIF 渲染 / loop tune / 结果评测
[6] Delivery Gate
    -> hard gate / deliver / internal / reject / manual_review
[7] Productization
    -> upload / collection / emoji / zip / public sync
[8] Optimization Loop
    -> feedback / rerender / baseline / manual score / prompt rollout
```

当前代码已经具备这个架构的大部分骨架，只是职责还需要进一步写清楚。

---

## 2. 每层职责边界

## 2.1 API / Task Admission 层

当前入口：`internal/handlers/video_jobs.go` -> `CreateVideoJob`

### 应负责

- 用户身份校验
- `source_video_key` 标准化
- 编辑参数合法性校验
- 格式能力限制
- `ffmpeg/ffprobe` 能力检查
- 源视频预探测
- 成本估算
- 积分冻结
- 写 `archive.video_jobs`
- 写 `public.video_image_jobs`
- 投递 Asynq

### 不应负责

- AI 判定
- 视频内容语义理解
- 具体候选选片
- 渲染策略细节

### 原因

入口层应该是**确定性、幂等、对账友好**的。AI 参与越少，任务提交越稳定。

---

## 2.2 Media Preprocess 层

当前主要落点：`processor.go` / `processor_pipeline.go`

### 应负责

- 下载源视频
- `ffprobe`
- 识别尺寸/时长/FPS
- 自动裁黑边/裁信箱区
- 基础片段合法性限制

### 输出

- 标准化视频元数据
- 裁切建议
- 可供候选召回使用的基础特征

### 设计原则

这一层只做**事实抽取**，不做“哪段最有价值”的业务判定。

---

## 2.3 Candidate Recall 层

当前核心：本地 `highlight scorer` + 后续候选筛选

### 应负责

- 从整段视频召回 Top-K 窗口
- 保证时间合法
- 初步去重叠
- 初步置信度过滤
- 为每个 candidate 写结构化特征

### 不应负责

- 直接做最终交付结论
- 直接控制 deliver/reject

### 最佳实践定位

这一层是**第一生产力**。

因为如果候选召回不到真正值得做 GIF 的窗口，后面 AI 再强也只是“在错误候选里挑相对较好的”。

---

## 2.4 AI Decision Layer

当前由三个角色组成：

- AI1 `Director`
- AI2 `Planner`
- AI3 `Judge`

这是现有架构里最需要写清职责边界的一层。

---

## 3. AI1 Director 的职责边界

当前代码：`requestAIGIFPromptDirective`

### 它应该做什么

- 定义任务目标：娱乐 / 传播 / 素材 / 新闻摘录
- 定义受众与场景
- 定义必须捕捉与必须规避内容
- 定义交付数量区间
- 定义时长偏好区间
- 定义 loop 偏好
- 定义质量权重
- 输出给 AI2 / AI3 使用的 brief

### 它不应该做什么

- 输出具体时间戳
- 对每个 candidate 逐条打分
- 直接裁决 deliver/reject

### 最佳实践总结

Director 是**目标函数生成器**，不是执行器。

#### 统一理解

如果把整个任务看成一次“剪辑 brief”，Director 就是这次任务的“甲方需求单”。

---

## 4. AI2 Planner 的职责边界

当前代码：`requestAIGIFPlannerSuggestion`

### 它应该做什么

- 在本地候选集合中做语义提名
- 综合 Director brief 与候选特征重新排序
- 给出结构化 proposal
- 输出提名理由、语义标签、standalone confidence、loop hint

### 它不应该做什么

- 直接操作原始视频全量扫描
- 跳过候选集合直接黑盒选片
- 直接变更账务/状态机
- 直接判定最终是否 deliver

### 最佳实践总结

Planner 是**候选集合上的语义排序器**。

#### 推荐输入边界

只喂给 Planner：

- Top-K candidates
- candidate 的结构化特征
- Director brief
- 视频 meta
- 交付上限与窗口约束

不要让 Planner 重新承担预处理与召回责任。

---

## 5. AI3 Judge 的职责边界

当前代码：`runAIGIFJudgeReview`

### 它应该做什么

- 对已经渲染出的成品 GIF 做语义复审
- 给出 `deliver / keep_internal / reject / need_manual_review`
- 给出诊断原因与下一步建议

### 它不应该做什么

- 覆盖技术 hard gate
- 直接替代渲染质量评测
- 直接抬升一个技术失败样本为 deliver

### 最佳实践总结

Judge 是**成品复审层**，不是最终技术门禁。

#### 非常关键的规则

Judge 只能：

- 降级交付结论
- 触发人工复审
- 给出语义解释

Judge 不能：

- 把硬失败样本强行升级为 deliver

---

## 6. Render & Evaluate 层职责

当前落点：

- `processor_pipeline.go`
- `processor_gif_render.go`
- `gif_evaluation.go`

### 应负责

- 多窗口渲染
- loop tune
- 产物大小控制
- 输出维度控制
- 质量评测
- 持久化 output / evaluation

### 不应负责

- 业务语义偏好判断
- 用户价值判断

### 最佳实践总结

这一层是**技术质量事实层**。

AI 可以参考它，但不应该篡改它。

---

## 7. Delivery Gate 层职责

当前落点：AI Judge + hard gate 组合

### 最终判定应来自两部分

1. **语义建议**：AI Judge
2. **技术约束**：hard gate

### 推荐优先级

```text
hard gate > judge recommendation > planner suggestion > director brief
```

### 这样做的好处

- 可解释
- 可回退
- 不会让 AI 越权
- 线上事故面更小

---

## 8. Productization 层职责

当前落点：

- `public_sync.go`
- 结果集合/表情/制品落库
- zip 打包
- 积分结算/成本核算

### 应负责

- 上传七牛
- 落库 collection / emoji / artifact
- 同步 `public.video_image_outputs`
- 生成 zip
- 写成本与积分结果

### 原则

产品化层只接收“已经确定好的产物和状态”，不要反向介入 AI 决策。

---

## 9. Optimization Loop 层职责

当前代码与表已经具备基础：

- `public.video_image_feedback`
- `ops.video_job_gif_rerank_logs`
- `ops.video_job_gif_baselines`
- `ops.video_job_gif_manual_scores`
- `gif_rerender.go`
- `ops.video_ai_prompt_templates`

### 这一层应负责

- 用户反馈学习
- 候选 rerank
- 历史样本回放
- 人工审核样本沉淀
- prompt A/B 与版本切换
- 提案重渲与回补

### 最佳实践定位

这一层不是“附加功能”，而是系统能不能持续变好的关键。

---

## 10. 角色边界矩阵

| 层/角色 | 可以做 | 不可以做 |
|---|---|---|
| API Admission | 校验、预探测、冻结积分、入队 | AI 语义判断 |
| Preprocess | probe、crop、基础特征提取 | 最终选片 |
| Candidate Recall | 召回候选、去重、多样性保留 | 最终交付判断 |
| AI1 Director | 生成 brief、目标权重 | 直接选时间戳 |
| AI2 Planner | 候选重排、proposal 提名 | 最终 deliver/reject |
| Render/Evaluate | 渲染、loop tune、评分 | 业务语义裁决 |
| AI3 Judge | 成品复审、语义建议 | 覆盖 hard gate |
| Delivery Gate | 最终判定与分流 | 反向修改原始评测事实 |
| Productization | 上传、入库、同步、结算 | 改写 AI 决策 |
| Optimization Loop | 反馈、重渲、实验、基线 | 直接污染生产状态机 |

---

## 11. 这份边界定义对当前代码的直接意义

如果后续改造，要尽量遵守下面三条：

### 11.1 不要让 AI1 和 AI2 职责混淆

Director 负责目标；Planner 负责窗口提名。

### 11.2 不要让 AI3 覆盖 hard gate

Judge 只给“语义建议”，hard gate 才是技术门槛。

### 11.3 不要让 rerender 重新跑整条重链路

重渲优先复用：

- 已有 proposal
- 已有 quality settings
- 已有 collection 与 public output 关系

这样成本最低，也最可控。

---

## 12. 最终建议

你当前架构方向是对的，接下来最重要的不是“加更多 AI”，而是：

- 把边界再写清楚
- 把实验与运营链路做强
- 把技术 gate 与业务 gate 对齐

这会让系统从“已经能跑”升级到“可以长期稳定迭代”。
