# 全量 HQ、不减张数前提下的 GIF 提速路线图（实施清单）

> 文档日期：2026-03-20  
> 对应上一版：`01-全量HQ不减张数提速路线图.md`

## 1. 实施目标
1. 保持全量识别高光、保持 HQ、不限制输出张数。
2. 将提速主战场从“降档/减产”转为“复用/调度/架构”。
3. 在不改产品目标前提下，优先压缩 render stage 耗时。

## 2. 代码改造清单（模块/函数级）

### 2.1 候选成本估算器升级（P0）

涉及文件：
- `internal/videojobs/processor_gif_candidates.go`
- `internal/videojobs/processor_gif_selection.go`

目标：
- 从仅估体积，升级为“体积 + 渲染耗时 + 成本单元”三指标。

建议改造：
1. 新增结构体：
```go
type gifCandidateCostEstimate struct {
    PredictedSizeKB    float64
    PredictedRenderSec float64
    CostUnits          float64
    ModelVersion       string
}
```
2. 将 `estimateGIFCandidateSizeKB(...)` 升级为：
```go
func estimateGIFCandidateCost(meta videoProbeMeta, candidate highlightCandidate, options jobOptions, qualitySettings QualitySettings) gifCandidateCostEstimate
```
3. 在 `feature_json` 持久化：
- `predicted_size_kb`
- `predicted_render_sec`
- `render_cost_units`
- `cost_model_version`

验收：
- 估算偏差明显收敛；用于调度时可稳定区分高成本窗口与低成本窗口。

---

### 2.2 引入 Bundle 规划器（P0/P1）

新增文件：
- `internal/videojobs/processor_gif_bundle_plan.go`

目标：
- 同一视频内多个窗口按时间邻近和参数相近进行分组，减少重复 decode/scale/fps。

建议数据结构：
```go
type gifRenderBundle struct {
    BundleID      string
    StartSec      float64
    EndSec        float64
    WindowIndexes []int
    CostUnits     float64
    Width         int
    FPS           int
    MaxColors     int
}
```

建议函数：
```go
func buildGIFRenderBundles(meta videoProbeMeta, tasks []animatedTask, qualitySettings QualitySettings) []gifRenderBundle
```

验收：
- 单 job 下多窗口任务的 ffmpeg 启动次数减少。

---

### 2.3 引入 Mezzanine 共享中间层（P1）

新增文件：
- `internal/videojobs/processor_gif_mezzanine.go`

目标：
- 在 bundle 维度生成归一化中间片段，供多窗口复用。

建议函数：
```go
type gifMezzanine struct {
    BundleID   string
    FilePath   string
    StartSec   float64
    EndSec     float64
    Width      int
    FPS        int
    PixelFmt   string
    SizeBytes  int64
}

func buildGIFMezzanine(ctx context.Context, sourcePath string, bundle gifRenderBundle, meta videoProbeMeta, qualitySettings QualitySettings) (gifMezzanine, error)
```

注意：
- 中间层是“可视无损复用层”，不是降质层。
- 临时文件生命周期由 job 上下文统一回收。

验收：
- 同 bundle 窗口渲染总时长明显下降。

---

### 2.4 引入成本感知调度器（P0/P1）

涉及文件：
- `internal/videojobs/processor_animated_task.go`
- `internal/videojobs/processor_persist_outputs.go`
- 新增 `internal/videojobs/processor_gif_scheduler.go`

目标：
- 由“固定 worker + 粗粒度 cap”升级为“成本令牌桶调度”。

建议接口：
```go
type gifRenderScheduleItem struct {
    Task      animatedTask
    CostUnits float64
    BundleID  string
}

type gifRenderScheduler interface {
    Run(ctx context.Context, items []gifRenderScheduleItem, fn func(context.Context, gifRenderScheduleItem) error) error
}
```

调度策略：
1. 总令牌上限 `gif_render_max_cost_units`。
2. 大窗口抢占更多令牌，小窗口可并行穿插。
3. 高成本任务避免全部堆在同一时刻。

验收：
- 不减张数条件下，render stage P50/P90 降低。

---

### 2.5 Loop Tune 与分析结果缓存复用（P1）

涉及文件：
- `internal/videojobs/processor_gif_render.go`
- 新增 `internal/videojobs/processor_gif_loop_cache.go`

目标：
- 避免每个窗口重复抽样 + 重复分析。

建议缓存键：
- `job_id + bundle_id + sample_profile_hash`

缓存内容：
- sample frame index
- motion/exposure/quality 统计
- loop closure 基础分

验收：
- loop tune 总耗时下降，效果稳定（回退率不恶化）。

---

### 2.6 HQ 子档（HQ 内部优化，不降 HQ）

涉及文件：
- `internal/videojobs/processor_animated_profile.go`
- `internal/videojobs/ai_gif_pipeline.go`

目标：
- 从“统一重参数 HQ”升级为“内容感知 HQ 子档”。

建议子档：
- `hq_portrait_reaction`
- `hq_action_burst`
- `hq_caption_meme`
- `hq_flat_background`

原则：
- 维持 HQ 视觉目标，不以统一参数上限作为 HQ 定义。

验收：
- 主观评分不下降，渲染成本下降。

## 7. 风险与回滚

## 7.1 风险点
1. bundling 处理不当可能引入窗口边界误差。
2. mezzanine 临时文件管理不当可能导致磁盘压力。
3. scheduler 令牌配置不合理可能导致局部饿死。
4. HQ 子档策略可能在个别内容上出现主观争议。

## 7.2 回滚策略
1. 所有新能力均需有 feature flag：
- `gif_scheduler_enabled`
- `gif_bundle_enabled`
- `gif_mezzanine_enabled`
- `gif_loop_cache_enabled`
- `gif_hq_subprofile_enabled`
2. 出现异常时可逐项关闭，回退到现有单窗口路径。

---

## 8. 验收标准（必须同时满足）
1. 不减张数。
2. 不降 HQ 定位。
3. render stage P50/P90 持续下降。
4. 失败率和重试率不升高。
5. 用户下载与留存相关指标不恶化。

---

## 9. 执行优先顺序（最终版）
1. 成本埋点 + 估算器 v2。
2. 成本感知调度器。
3. bundling + mezzanine。
4. loop 缓存复用。
5. HQ 子档策略。
6. 独立 render worker 池。

该顺序保证在“全量 HQ、输出不减”的前提下，优先获取最大提速收益。
