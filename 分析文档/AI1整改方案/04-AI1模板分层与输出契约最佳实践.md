# AI1 模板分层与输出契约最佳实践

## 1. 先说判断：你当初这套设计思路是对的

你现在强调的三件事：

1. **可编辑层**
2. **固定层**
3. **输出给 AI2 的输出契约**

我看完当前代码后，判断是：

> 这三层设计方向本身是对的，而且非常适合你现在这条 `AI1 -> AI2 -> worker&打分 -> AI3` 链路。

因为它本质上是在把 Prompt 从“写给一个模型看的话术”，升级为：

- **固定层：协议层 / Protocol Layer**
- **可编辑层：策略层 / Policy Layer**
- **输入 JSON：数据层 / Data Layer**
- **输出契约：模型间协议层 / Inter-model Contract**

这是从“单模型 Prompt”走向“多模型协作系统”的正确方向。

## 3. 三层的职责边界，必须明确

这是最佳实践里最关键的一点。

---

## 4. 固定层的职责：负责“协议、边界、底线”

固定层不应该写太多具体业务策略，它应该负责的是那些：

- **每次都成立**
- **不能让运营随意改**
- **一旦改坏会影响系统稳定**
- **属于 AI1 基础行为规范**

的内容。

### 4.1 固定层应该包含什么

我建议固定层只负责 6 类内容：

#### A. 角色定位

例如：

- 你是谁：AI1 / Director
- 你处于哪个阶段：AI2 前置 briefing 阶段
- 你的目标是什么：不是直接剪辑，而是输出给 AI2 的结构化任务合同

#### B. 输入理解规则

例如：

- 视频是主感知源
- 文本 JSON 是任务合同，不是视频内容摘要
- 模型不能忽略视频而只看 JSON
- 模型不能凭空臆测视频里不存在的内容

#### C. 决策框架

例如：

- 必须综合考虑：
  - 业务目标
  - 独立成图能力
  - 高光价值
  - 画面清晰度
  - 循环潜力
  - 风险因素

#### D. 不确定性处理规则

例如：

- 当视频信号弱时，应选择更稳妥的窗口策略
- 当找不到强循环点时，不允许为追求 loop 硬编
- 当内容强依赖上下文时，应降低 standalone 判断

#### E. 输出契约硬规则

例如：

- 只能输出 JSON
- 外层必须是 `{"directive":{...}}`
- 不能输出 markdown
- 不能输出解释文字
- 不能新增未定义包装层

#### F. 契约字段的基础合法性

例如：

- `clip_count_max >= clip_count_min`
- `duration_pref_max_sec > duration_pref_min_sec`
- `quality_weights` 和接近 1
- 所有评分在 `[0,1]`

这类约束现在其实已经写在默认 fixed prompt 中了：

- `internal/videojobs/ai_gif_pipeline.go:58`

### 4.2 固定层不应该包含什么

以下内容不建议写进固定层：

#### 1）阶段性运营策略

例如：

- 最近成本低，可以多出候选
- 最近优先社交传播
- 最近强调情绪峰值

这些都应该是 editable 层的内容。

#### 2）短期实验口径

例如：

- 本周要测试“更激进的多候选”
- 本月要提高封面型素材比例

这也不应该放固定层。

#### 3）用户需求槽位

未来如果你要支持“用户自然语言描述需求”，这部分绝不能写死在 fixed 层。

### 4.3 固定层最佳实践总结

一句话：

> **固定层负责“系统级不可变规则”，不负责“业务级可变策略”。**

## 5. 可编辑层的职责：负责“运营策略、业务偏好、未来扩展口子”

你对可编辑层的理解也是对的：

> 它本质上是运营和模型在 AI1 阶段的第一轮业务交互。

它应该表达的不是“协议”，而是：

- 我是谁
- 我为什么要这类素材
- 我想得到什么结果
- 我偏向什么，不要什么
- 在成本/质量之间如何取舍
- 如果以后用户有自然语言需求，要怎样挂进来

### 5.1 可编辑层为什么一定要可编辑

原因非常明确：

#### 1）成本条件会变

比如：

- token 成本降低
- 可以放宽候选窗口数量
- 可以更激进探索更多候选

#### 2）业务目标会变

比如：

- 从“传播型 GIF”转向“可做封面图的静态时刻”
- 从“娱乐反应”转向“商品展示完整动作”

#### 3）用户意图会变

未来你要支持用户自然语言需求，这个层一定要给用户意图留位置。

#### 4）运营语言会变

运营团队的理解框架、表达方式、关注点，本来就是动态变化的。

### 5.2 但可编辑层不等于“完全自由写 Prompt”

这是最佳实践里特别重要的一条。

我不建议长期让可编辑层只是一个完全自由的大文本框。

原因：

1. 容易把输出契约改坏
2. 容易把固定层底线冲掉
3. 容易出现多人版本漂移
4. 容易堆砌话术，降低模型稳定性
5. 不利于后续支持“用户自然语言需求槽位”

所以我建议：

> **可编辑层的正确长期形态，不是“纯文本自由写”，而是“字段化配置 + 渲染成 Prompt”。**

这点你们当前系统其实已经留了一个很好的口子：

- `VideoAIPromptTemplate.TemplateJSONSchema`
- `internal/models/video_jobs.go:599`

但目前运行时还没有真正把它用起来。

这意味着：

> 你们系统现在已经有“把 editable 从纯文本升级为结构化配置”的基础。

## 6. 可编辑层推荐字段

我建议把 editable 层拆成“字段化策略配置”，由系统再渲染成一段 Prompt。

下面是推荐字段。

### 6.1 身份与任务定义

| 字段 | 类型 | 说明 |
|---|---|---|
| `operator_identity` | string | 运营身份/视角，例如“社交传播运营”“电商内容运营” |
| `business_scene` | string | 当前业务场景，如 `social_spread / ecommerce / archive` |
| `asset_goal` | string | 当前要产出的资产类型，如 `gif_highlight / cover_image / keyframe_pack` |
| `delivery_goal` | string | 对结果的定义，如“独立传播”“可点击封面”“步骤完整” |

### 6.2 成功标准

| 字段 | 类型 | 说明 |
|---|---|---|
| `success_definition` | string[] | 成功素材应具备的特征 |
| `must_capture_bias` | string[] | 希望优先抓取的内容模式 |
| `avoid_bias` | string[] | 应尽量避免的内容模式 |

### 6.3 成本与候选策略

| 字段 | 类型 | 说明 |
|---|---|---|
| `cost_mode` | enum | `strict / balanced / aggressive` |
| `candidate_strategy` | enum | `conservative / balanced / exploratory` |
| `candidate_count_bias` | enum | `low / medium / high` |
| `diversity_preference` | enum | `low / medium / high` |

### 6.4 质量偏好

| 字段 | 类型 | 说明 |
|---|---|---|
| `clarity_priority` | enum | `low / medium / high` |
| `loop_priority` | enum | `low / medium / high` |
| `standalone_priority` | enum | `low / medium / high` |
| `emotion_priority` | enum | `low / medium / high` |

### 6.5 风险与限制

| 字段 | 类型 | 说明 |
|---|---|---|
| `risk_tolerance` | enum | `low / medium / high` |
| `subtitle_dependency_tolerance` | enum | `low / medium / high` |
| `quality_floor_note` | string | 对模糊、遮挡、抖动等底线说明 |

### 6.6 为未来用户自然语言需求预留

| 字段 | 类型 | 说明 |
|---|---|---|
| `user_request_passthrough` | string | 未来直接挂用户自然语言需求 |
| `extra_business_note` | string | 运营补充说明 |

### 6.7 最关键的治理规则

这些字段可以编辑，但不应该允许 editable 层直接定义：

- JSON 外层 key 名
- 输出包装结构
- 必填字段名
- 解析规则
- 输出是否允许 markdown
- 字段合法范围

也就是说：

> **可编辑层可以改“值倾向”，不能改“协议骨架”。**

## 7. 可编辑层最佳实践：不是让运营直接写一大段，而是字段化后渲染

### 7.1 推荐形态

建议未来演进为：

#### 存储层

存：

- `template_text`：模板渲染骨架
- `template_json_schema`：字段 schema
- `metadata`：版本、用途、适用场景

#### 编辑层

运营填写：

```json
{
  "operator_identity": "社交传播运营",
  "business_scene": "social_spread",
  "asset_goal": "gif_highlight",
  "delivery_goal": "standalone_shareable",
  "success_definition": ["高光独立成立", "表情/动作完整", "适合转发"],
  "cost_mode": "balanced",
  "candidate_strategy": "balanced",
  "diversity_preference": "medium",
  "clarity_priority": "high",
  "loop_priority": "medium",
  "subtitle_dependency_tolerance": "low",
  "user_request_passthrough": ""
}
```

#### 渲染层

系统把这些字段渲染为可读 Prompt 片段，再拼到 fixed 层后面。

### 7.2 这样做的好处

1. 运营可编辑
2. 可审计
3. 可版本化
4. 可回放
5. 可做 AB 实验
6. 可安全接入未来用户自然语言需求

## 8. 推荐的可编辑层 Prompt 内容组织

如果你现在还不想一下子升级为完整字段渲染系统，我建议 editable 至少按下面 6 段组织内容：

### 段 1：运营身份

说明：

- 我是谁
- 我代表什么业务目标

### 段 2：任务目标

说明：

- 这次我要什么类型的素材
- 结果交付偏向什么

### 段 3：成功标准

说明：

- 好结果应该长什么样
- 优先抓什么

### 段 4：限制与避坑

说明：

- 尽量避免什么片段
- 对字幕依赖、模糊、抖动的容忍度

### 段 5：成本策略

说明：

- 低成本时更保守
- 低成本约束放松时更探索

### 段 6：未来用户需求槽位

说明：

- 如有用户自然语言需求，应优先融入判断

## 9. 固定层的最佳实践内容结构

我建议固定层不是一整段平铺，而是明确拆成以下模块。

### 9.1 模块 A：角色与阶段

例如：

- 你是 AI1 / Prompt Director
- 你不直接做剪辑
- 你输出的是给 AI2 的结构化任务合同

### 9.2 模块 B：输入解释规则

例如：

- 视频是感知主源
- JSON 是任务合同
- 如果视频和 JSON 有冲突，应优先以视频事实为准，但要保留 JSON 中的业务目标

### 9.3 模块 C：判断维度

例如：

- 独立成图能力
- 语义高光
- 清晰度
- 循环潜力
- 风险规避
- 成本意识

### 9.4 模块 D：冲突处理

例如：

- 当运营强调多候选，但视频质量弱时，仍要保守
- 当运营强调低成本时，应优先少而准，而不是多而散
- 当找不到理想 loop 时，应退而求其次选择 standalone 强的片段

### 9.5 模块 E：输出规则

例如：

- 只输出 JSON
- 不输出 markdown
- 不输出额外解释
- 结构必须稳定

### 9.6 模块 F：输出契约尾段

这是我非常建议保留的做法：

> **固定层的最后必须再追加一段“契约尾段”**，专门负责锁死输出结构。

你们当前代码其实已经这么做了：

- `internal/videojobs/ai_gif_pipeline.go:849`

这很对。

我的建议是把它显式命名为：

- `fixed_core_prompt`
- `editable_policy_prompt`
- `fixed_contract_tail`

这样语义会更清楚。

## 10. 输出契约的本质：它不是“回答”，而是“交接协议”

这是你这个设计里最重要的一点。

AI1 输出给 AI2 的内容，不应该理解成：

- “AI1 对任务的自然语言总结”

而应该理解成：

- “AI1 对 AI2 下发的结构化工作单”

所以输出契约的目标不是“人看起来顺”，而是：

1. AI2 容易理解
2. 程序容易解析
3. 字段稳定可版本化
4. 未来能逐步转成硬消费

---

## 11. 当前输出契约的优点与不足

### 11.1 当前优点

当前已有字段其实已经有不错基础：

- `business_goal`
- `must_capture`
- `avoid`
- `clip_count_*`
- `duration_pref_*`
- `loop_preference`
- `risk_flags`
- `quality_weights`
- `directive_text`

这些字段让 AI1 已经不再只是输出一句话，而是输出结构化意图。

### 11.2 当前不足

但从“AI1 给 AI2 下工作单”的角度，它还不够完整。

主要问题：

#### 1）缺少协议版本

建议新增：

- `contract_version`

#### 2）缺少任务分层

当前字段是平铺的，建议拆成：

- `task`
- `selection_policy`
- `ranking_policy`
- `fallback_policy`

#### 3）缺少 AI2 的执行视角字段

AI2 真正关心的是：

- 我要找几类候选
- 每类候选时间窗口怎么控制
- 优先级怎么排
- 允许多样性还是要集中
- 遇到弱信号怎么办

这些当前还比较隐含。

#### 4）`directive_text` 重要，但不能成为主载体

我建议：

- 保留 `directive_text`
- 但它只能做“补充语义摘要”
- 真正关键约束必须结构化

---

## 12. 输出契约最佳实践：一半“机器字段”，一半“模型摘要”

我建议 AI1 输出契约遵循下面原则：

### 12.1 必须有稳定包装层

保持：

```json
{"directive":{...}}
```

不建议让 editable 层动这个。

### 12.2 必须有版本号

例如：

- `contract_version: "ai1_to_ai2_v2"`

### 12.3 字段分硬约束与软语义

#### 硬约束字段

程序未来应该可直接消费的字段，例如：

- count
- duration
- overlap
- diversity
- priority
- fallback policy

#### 软语义字段

模型理解类字段，例如：

- `must_capture`
- `avoid`
- `directive_text`
- `search_hints`

### 12.4 结构化优先，短文本兜底

建议：

- 70% 指令信息来自结构化字段
- 30% 来自 `directive_text`

而不是反过来。

### 12.5 不能过深嵌套

输出契约嵌套太深会降低模型稳定性，建议控制在 2~3 层。

### 12.6 枚举优于自由文本

对于会影响下游稳定性的字段，优先使用枚举：

- `diversity_preference`
- `overlap_tolerance`
- `subtitle_dependency_tolerance`
- `fallback_policy.*`

## 14. 这版输出契约为什么更适合 AI2

### 14.1 task 层

告诉 AI2：

- 这次到底在做什么资产
- 服务什么业务场景
- 交付目标是什么

### 14.2 selection_policy 层

告诉 AI2：

- 候选窗口找多少
- 时长怎么卡
- 要不要多样性
- 候选之间允许多重叠

### 14.3 ranking_policy 层

告诉 AI2：

- 排名优先级怎么排
- quality_weights 怎么理解
- 对字幕依赖容忍度如何

### 14.4 fallback_policy 层

告诉 AI2：

- 找不到理想峰值怎么办
- 找不到理想 loop 怎么办
- 质量和数量冲突时优先什么

这部分是当前契约最缺的一层，但实际上很重要。

## 15. 固定层、可编辑层、输出契约之间的最佳实践关系

这三者的关系，我建议你以后统一成下面这条原则：

### 固定层负责：

- **字段定义**
- **角色定义**
- **输出规则**
- **冲突处理底线**

### 可编辑层负责：

- **值偏好**
- **业务策略**
- **成本策略**
- **未来用户意图**

### 输出契约负责：

- **把 AI1 的理解转成 AI2 能读懂的结构化工作单**

一句话总结：

> **固定层定骨架，可编辑层定偏好，输出契约定交接。**

---

## 16. 最佳实践中的“红线”

### 16.1 绝对不要让 editable 层改这些

- 外层 JSON 包装
- `directive` key
- 字段名
- 必填字段
- JSON-only 规则
- no-markdown 规则
- 字段合法性范围

### 16.2 绝对不要把这些长期放在 fixed 层

- 临时业务策略
- 成本松紧调整
- 实验口径
- 用户自然语言需求槽位

### 16.3 输出契约不要变成纯自然语言

如果 AI1 输出最后退化成一长段 prose，即使 AI2 是模型，也会降低稳定性。

必须坚持：

- 结构化字段为主
- 摘要文本为辅

## 17. 结合你们当前系统，我的实践建议

### 17.1 短期（马上可做）

1. 保留当前 fixed + editable + contract tail 拼装方式
2. 把 editable 从“自由大文本”逐步整理成分段模板
3. 明确写规则：
   - editable 不得改输出 schema
4. 给 AI1 输出契约加 `contract_version`
5. 给输出契约补：
   - `selection_policy`
   - `ranking_policy`
   - `fallback_policy`

### 17.2 中期（最值得做）

1. 真正启用 `template_json_schema`
2. editable 改成“字段化配置 + 渲染”
3. 后台运营编辑页改成表单式，不再只是大文本框
4. 保留一个 `user_request_passthrough` 槽位

### 17.3 长期（体系化）

1. AI1 输出契约部分字段变成程序硬消费
2. AI2 不只“读 prompt”，还“读结构”
3. worker / AI3 也逐步接这份合同

## 18. 最终结论

你的这套想法，我给一个很明确的结论：

> **方向是对的，而且应该继续强化，不是削弱。**

尤其是下面这三点要坚持：

### 第一，固定层必须存在

因为它不是“方便”，它是整个系统的协议底座。

### 第二，可编辑层必须存在，而且以后要从文本升级为字段化

因为运营策略、成本策略、用户意图，本来就是可变的。

### 第三，输出契约必须存在，而且要把“给 AI2 的工作单”设计得更结构化

因为这不是聊天，而是多模型协作。

如果只用一句话概括这份方案，就是：

> **AI1 不是在回答问题，而是在给 AI2 下发结构化任务合同。**

而固定层、可编辑层、输出契约，分别就是这份合同的：

- 制度
- 策略
- 交接格式
