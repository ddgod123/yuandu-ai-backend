## 2. 必补 P0（开工前要收口）

### P0-1 幂等与并发契约

需要在文档中补明确规则：

- 人审接口幂等键（建议：`job_id + output_id + operator_id + request_id`）。
- 批量重渲幂等键（建议：`job_id + sorted_proposal_ids + strategy + request_id`）。
- 同一 Job 并发重渲冲突策略（拒绝/排队/覆盖其一，必须固定）。

### P0-2 状态机不可逆规则

需要补“合法转移表”：

- 主 stage（queued -> preprocessing -> ... -> done/failed/cancelled）。
- GIF sub-stage（briefing/planning/scoring/reviewing）。
- 终态是否允许再进入重渲补偿链路（建议允许补偿，不回滚主终态）。

### P0-3 指标口径字典

需要固定分母和窗口口径：

- 全部监控统一时区与窗口定义（建议 Asia/Shanghai + rolling 24h）。
- `no_data` 与 `critical` 分离判定。
- 每个指标提供 SQL 口径注释（至少在脚本层）。

### P0-4 成本可复算约束

`ops.video_job_ai_usage` 建议强制保留：

- pricing_version
- pricing_source_url
- usd/cny 汇率快照时间
- input/output/cached/image/video/audio 的 token/计费口径

目标：支持财务复盘与事后对账。

- Prompt 管理：`/api/admin/video-jobs/ai-prompt-templates*`

## 4. 建议实施顺序（可直接进入开发）

### 阶段 A（本周）

1. 先补 `gif-audit-chain` 聚合查询与详情页展示。
2. 同步补 `policy_version / experiment_bucket / hard_gate_reason_codes` 的统一落点。
3. 增加 Judge 与 Hard Gate 冲突指标（监控可见）。

### 阶段 B（下周）

1. 上线人工审核决策接口。
2. 人审决策入库并打审计日志。
3. 后台页支持按状态筛选与人工决策回放。

### 阶段 C（第二周后半）

1. 批量重渲接口（可选）。
2. 成本分层策略开关（light/standard/hq）灰度。
3. 周期性报表（质量 + 成本 + 反馈）联动。

## 5. 本轮不建议立即做

1. 先不上新主表（`ops.video_job_experiment_runs` / `ops.video_job_delivery_metrics_daily`），先用现有表聚合验证价值。
2. 先不引入更重模型/更复杂多模态输入，优先把候选质量与链路治理跑稳。
3. 先不同时改 Prompt + 候选召回 + Gate 阈值，保持单变量迭代。

---

## 6. 验收标准（最小可用）

1. 任意 job 可在后台一页串起全链路。
2. 能明确区分：AI 失败 / 技术失败 / 业务降级。
3. Judge 与 Hard Gate 冲突可量化并趋势化。
4. 人审决策可追溯（谁在何时改了什么）。
5. 单任务成本可复算、可解释。
