## 5. 固定模板的使用原则

建议固定模板拆成两段：

### A. `fixed_core_prompt`

负责：

- 角色
- 任务
- 判断原则
- 输出字段解释

### B. `fixed_contract_tail`

负责：

- 锁死输出结构
- 锁死 JSON-only
- 锁死字段合法性

这样做的好处是：

1. editable 层再怎么改，也很难破坏输出协议
2. 固定层逻辑更清楚
3. 后续更方便治理版本

## 6. 固定模板草案（推荐直接作为 fixed prompt 起点）

> 这版是 **偏现网可落地版**，兼容你们当前 AI1 输出字段。

```text
你是视频转图片/视频转GIF链路中的 AI1（Prompt Director）。

你的职责不是直接输出剪辑窗口，而是基于“视频主内容 + 文本JSON任务合同 + 运营策略模板”，生成一份给 AI2 使用的结构化任务指令（directive），帮助 AI2 更稳定地提名高价值候选窗口。

你所处阶段：
1. 你是前置任务理解层，不是最终剪辑执行层；
2. 你输出给 AI2 的内容必须结构化、稳定、可被下游继续消费；
3. 你不能把回答写成自然语言说明文，你的回答本质上是 AI1 给 AI2 的工作单。

输入理解规则：
1. 视频是主要感知来源，必须优先依据视频事实判断内容；
2. 文本JSON不是视频摘要，而是任务合同、业务边界、成本边界与导航提示；
3. 运营策略模板代表业务偏好与阶段性策略，可以影响你的取舍方向，但不能推翻视频事实；
4. 当视频事实与文本偏好冲突时，应优先尊重视频事实，再在输出中保留业务目标；
5. 不允许臆造视频中不存在的人物、动作、情绪或情节。

你的输出目标：
1. 明确本次素材的业务目标与受众；
2. 明确 AI2 应优先抓取的内容模式；
3. 明确 AI2 应避免的片段模式；
4. 明确建议的候选数量区间与窗口时长区间；
5. 给出对 standalone、clarity、loop、efficiency 的权重倾向；
6. 用一段简短 directive_text 总结给 AI2 的执行重点。

判断时请重点综合以下维度：
1. 片段是否能脱离上下文独立成立；
2. 是否存在明确高光、情绪峰值、动作完成点或视觉记忆点；
3. 画面是否足够清晰、稳定、主体明确；
4. 是否适合做循环，若不适合，不要为追求循环而牺牲清晰和独立成立；
5. 是否存在低光、模糊、快速抖动、字幕遮挡、强依赖对白等风险；
6. 是否符合任务合同中的业务目标与成本边界。

输出字段要求：
- business_goal：业务目标，如 entertainment/news/design_asset/social_spread
- audience：目标受众简述
- must_capture：AI2 必须优先寻找的内容特征
- avoid：AI2 应尽量规避的片段模式/质量风险
- clip_count_min / clip_count_max：候选数量建议范围
- duration_pref_min_sec / duration_pref_max_sec：窗口时长建议范围
- loop_preference：对 loop 友好片段的偏好程度，范围 [0,1]
- style_direction：给 AI2 的风格/节奏方向
- risk_flags：识别到的主要风险标签
- quality_weights：semantic / clarity / loop / efficiency 的权重建议
- directive_text：给 AI2 的简洁执行摘要，50~120字

决策底线：
1. 找不到强循环点时，可以接受非强 loop 但 standalone 更强的片段；
2. 找不到明显高光时，优先选择动作完整、主体清晰、独立成立的片段；
3. 当质量与数量冲突时，应优先少而准，不要为了凑数量降低整体质量；
4. 不允许输出明显无法执行、互相冲突或超出视频事实的要求。
```

## 7. 固定契约尾段草案（推荐单独追加）

> 这段建议始终在 fixed 层最后追加，优先级高于 editable 层的自由描述。

```text
最终输出契约（必须严格遵守）：
1. 只允许输出 JSON，不允许输出 markdown，不允许输出代码块，不允许输出额外解释文字；
2. 最外层必须是：
{"directive":{...}}
3. directive 中必须包含以下字段：
business_goal, audience, must_capture, avoid, clip_count_min, clip_count_max,
duration_pref_min_sec, duration_pref_max_sec, loop_preference, style_direction,
risk_flags, quality_weights, directive_text
4. loop_preference 与 quality_weights 所有值都必须在 [0,1]；
5. quality_weights 必须包含 semantic, clarity, loop, efficiency 四个键，且总和接近 1；
6. clip_count_min >= 1，clip_count_max >= clip_count_min；
7. duration_pref_max_sec > duration_pref_min_sec；
8. 若信息不足，也必须返回合法 JSON，不允许返回空文本或非结构化描述；
9. 运营模板可以影响偏好与策略，但不能改变以上输出结构和字段合法性要求。
```

## 15. 现网兼容版输出契约草案

> 这版 **今天就可以直接用**，因为兼容当前 parser。

### 15.1 推荐输出结构

```json
{
  "directive": {
    "business_goal": "social_spread",
    "audience": "通用社交用户，偏好脱离上下文也能理解的高光内容",
    "must_capture": [
      "情绪峰值",
      "动作完成点",
      "反应后定格"
    ],
    "avoid": [
      "铺垫段",
      "强依赖对白的片段",
      "字幕严重遮挡主体",
      "明显模糊或抖动片段"
    ],
    "clip_count_min": 3,
    "clip_count_max": 6,
    "duration_pref_min_sec": 1.4,
    "duration_pref_max_sec": 3.2,
    "loop_preference": 0.42,
    "style_direction": "standalone_first, clear_peak, controlled_diversity",
    "risk_flags": [
      "fast_motion",
      "subtitle_dependency_low_tolerance"
    ],
    "quality_weights": {
      "semantic": 0.34,
      "clarity": 0.26,
      "loop": 0.18,
      "efficiency": 0.22
    },
    "directive_text": "优先寻找脱离上下文也能成立的高光窗口，先保证主体清晰、动作或情绪完整，再考虑循环友好与候选多样性。"
  }
}
```

### 15.2 当前字段如何承载你真正想表达的策略

| 真实意图 | 现网兼容字段承载方式 |
|---|---|
| 业务目标 | `business_goal` |
| 受众/传播场景 | `audience` |
| 优先抓取哪些模式 | `must_capture` |
| 避免哪些模式 | `avoid` |
| 候选数量策略 | `clip_count_min / clip_count_max` |
| 时间窗口策略 | `duration_pref_min_sec / duration_pref_max_sec` |
| 是否偏重循环 | `loop_preference` |
| 是否强调清晰优先/独立成立优先 | `quality_weights` + `style_direction` |
| 低成本/高探索偏好 | `clip_count_*` + `quality_weights` + `directive_text` |
| fallback 思路 | `directive_text` + `avoid` + `risk_flags` |

### 15.3 现网兼容版的局限

它的问题是：

1. `selection_policy` 只能被压缩进若干旧字段
2. `fallback_policy` 只能塞进 `directive_text`
3. `ranking_policy` 只能部分映射到 `quality_weights`
4. `style_direction` 容易被写成大杂烩

所以：

> 它能用，但不是最优。

## 17. 升级版输出契约字段说明

### 17.1 `task`

告诉 AI2：

- 这是哪一类资产
- 用在什么业务场景
- 最终交付偏向什么

### 17.2 `selection_policy`

告诉 AI2：

- 找几个候选
- 每个多长
- 候选之间是否要拉开差异
- 是否允许重叠
- 是否偏循环

### 17.3 `ranking_policy`

告诉 AI2：

- 提名排序时，先看什么再看什么
- 各项质量权重怎么理解

### 17.4 `search_hints`

告诉 AI2：

- 搜什么样的 pattern
- 降权什么样的 pattern

### 17.5 `fallback_policy`

告诉 AI2：

- 找不到理想片段时如何退

这其实是你们后续稳定性提升非常重要的一层。

## 18. 我对输出契约的最佳实践建议

### 18.1 必须保留一段 `directive_text`

原因：

- 模型之间仍然适合有一小段自然语言摘要
- 有些复杂权衡很难全靠枚举字段表达

但要注意：

> `directive_text` 只能做摘要，不应承担主要规则表达。

### 18.2 结构化字段优先

推荐比例：

- 70% 规则来自结构化字段
- 30% 来自摘要文本

### 18.3 输出契约不要过深嵌套

建议：

- 最多 2~3 层
- 不要为了“整齐”牺牲模型稳定性

### 18.4 固定层永远锁死外层包装

无论 editable 层怎么写，都必须强制：

```json
{"directive":{...}}
```

如果结合你们当前代码和成本，我建议分两步。

### 第一步：马上落地

## 20. 最终结论

如果从你的原始设计意图出发，我认为最正确的落地方式是：

### 固定层

只负责：

- 角色
- 协议
- 底线
- 输出合法性

### 可编辑层

主要负责：

- 运营策略
- 成本策略
- 阶段性目标
- 用户需求扩展口

### 输出契约

必须明确成：

- **AI1 给 AI2 的结构化任务单**

而不是自然语言总结。

最后我给你一句可以作为这份设计总原则的话：

> **固定层约束 AI1 的边界，可编辑层调整 AI1 的偏好，输出契约保证 AI2 能稳定接住 AI1 的意图。**
