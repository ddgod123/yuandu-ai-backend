# 实施路线与改造清单

## 1. 总体原则

实施顺序建议遵循：

1. **先补观测，再改策略**
2. **先补审计链，再改 Prompt**
3. **先增强候选质量，再增加模型复杂度**
4. **先小流量试验，再全量放开**

原因很简单：

- 没有观测，很难知道优化是否有效
- 没有审计链，出了问题无法归因
- 候选召回弱，后续 AI 再强也救不回来
- 大流量直接改，很容易事故

---

## 2. Phase 0：补观测与审计（建议 1~2 周）

## 2.1 目标

让任意一个任务都能完整回答：

- 候选从哪里来
- proposal 为什么这样排
- output 为什么被 deliver/reject
- AI 和 hard gate 谁最终决定了结果

## 2.2 具体改造点

### 代码模块

- `internal/videojobs/processor_pipeline.go`
- `internal/videojobs/ai_gif_pipeline.go`
- `internal/videojobs/public_sync.go`
- `internal/videojobs/gif_rerender.go`

### 建议动作

1. 统一补 `policy_version / experiment_bucket`
2. 统一补 `hard_gate_reason_codes`
3. 统一补 `director/planner/judge prompt version` 快照
4. 统一补 `delivery_decision_summary`
5. 增加完整 `gif audit chain` 查询聚合逻辑

## 2.3 验收标准

- 任意 job 都能串出完整链
- 能区分 AI 失败、technical fail、业务降级
- 版本信息可追踪

---

## 3. Phase 1：补后台接口与运营可视化（建议 1~2 周）

## 3.1 建议优先实现

### 接口

- `GET /api/admin/video-jobs/:id/gif-audit-chain`
- `POST /api/admin/video-jobs/:id/gif-review-decisions`
- `POST /api/admin/video-jobs/:id/rerender-gif/batch`（可选）

### 后台页

- Job detail 审计页
- Prompt version 对比页
- Delivery / Cost tab

## 3.2 代码可能落点

- `internal/handlers/admin_video_jobs.go`
- 新增聚合 handler 或 detail handler
- 复用现有 `admin_video_job_detail.go` 思路

## 3.3 验收标准

- 运营能独立定位 case
- 运营能发起人工审核与重渲
- 后端不需要直接查多张表才能解释一次失败

---

## 4. Phase 2：增强候选召回与 Planner 输入质量（建议 2~4 周）

## 4.1 目标

提高 top proposal 的命中率，降低 Judge 打回与 rerender 率。

## 4.2 建议动作

1. 强化 candidate `feature_json`
2. 加入候选多样性约束
3. 统一 candidate 输入结构给 Planner
4. 把 `scene_score / motion / clarity_estimate / loop_estimate` 纳入 Planner 上下文
5. 优化 `resolveOutputClipWindows` 的筛选策略顺序

## 4.3 可能涉及模块

- `processor.go`
- `processor_pipeline.go`
- `ai_gif_pipeline.go`
- `gif_evaluation.go`

## 4.4 验收标准

- proposal top1 accept rate 提升
- candidate 扎堆现象下降
- Judge reject rate 下降
- 单任务重渲率下降

---

## 5. Phase 3：校准 Judge 与 hard gate（建议 1~2 周）

## 5.1 目标

让 Judge 和 technical gate 不打架。

## 5.2 建议动作

1. 固化最终决策优先级
2. 给 review metadata 增加 hard gate before/after 快照
3. 建立冲突样本集：
   - Judge deliver 但 hard gate reject
   - Judge reject 但用户反馈好
4. 建立人工边界样本复核机制

## 5.3 可能涉及模块

- `ai_gif_pipeline.go`
- `admin_video_jobs_gif_health.go`
- `ops.video_job_gif_manual_scores` 相关导出/管理逻辑

## 5.4 验收标准

- hard gate / judge 冲突率下降
- manual review 更集中在真正边界样本
- deliver 稳定性提升

---

## 6. Phase 4：做成本分层策略（建议 1~2 周）

## 6.1 目标

在不明显损伤质量的前提下降低平均 AI 成本。

## 6.2 建议动作

1. 按视频长度分层
2. 按候选置信度分层
3. 按用户价值分层
4. 对短视频优先走轻链路
5. 对高价值任务才全开 AI1+AI2+AI3

## 6.3 可能涉及模块

- `processor_pipeline.go`
- `quality_settings.go`
- 新增轻量策略选择器，如 `resolveGIFPipelineMode()`

## 6.4 验收标准

- avg ai cost per job 下降
- avg cost per deliver 下降
- deliver rate 无明显恶化

---

## 7. Phase 5：把人工审核和重渲做成正式运营闭环（建议 2~3 周）

## 7.1 目标

让人工审核与重渲从“事故补锅工具”升级为“优化工具”。

## 7.2 建议动作

1. 支持人工确认 deliver / keep_internal / reject
2. 支持按 proposal 批量试渲
3. 支持重渲策略模板
4. 支持对历史 case 做小规模回放
5. 样本自动回流到 manual score / baseline 分析

## 7.3 验收标准

- 人工审核产出能被沉淀
- rerender 翻盘能被量化
- 运营能独立推动一轮优化闭环

---

## 9. 风险点与规避建议

## 9.1 风险：同时改太多层，无法归因

### 规避

一轮只改一类变量：

- 先改 prompt
- 或先改 recall
- 或先改 hard gate

不要一起改。

---

## 9.2 风险：manual review 暴增

### 规避

- Judge 改动保守上线
- hard gate 边界先灰度
- 先看冲突率，再放量

---

## 9.3 风险：成本下降但质量也下降

### 规避

成本策略必须同时观察：

- deliver rate
- 正反馈率
- rerender 翻盘率

不能只看 token 成本。

---

## 9.4 风险：后台可视化不足，运营无法跟进

### 规避

Phase 1 必须优先做审计链与 review 接口。

---

## 10. 推荐里程碑

## 里程碑 M1

- 完成 audit chain
- 完成版本字段补齐
- 能在后台完整解释一个 case

## 里程碑 M2

- 完成人工审核闭环
- 完成批量重渲基础能力
- 能对历史 case 做小规模优化实验

## 里程碑 M3

- 完成候选输入增强
- proposal 命中率明显提升
- judge/hard gate 冲突下降

## 里程碑 M4

- 完成成本分层
- 高质量任务与普通任务分流
- 单 deliver 成本下降

---

## 11. 最终实施建议

如果你要开始真正动代码，我建议最先落地的是这 3 件事：

1. **补一条完整 `gif audit chain`**
2. **补 `manual review` 的后台定结论能力**
3. **统一补齐各阶段的 version / policy / hard gate 快照**

因为这 3 件事一旦做完，后面无论是优化 prompt、优化召回、优化成本，都会容易很多。

---

## 12. 最后一条建议

这套系统下一步最应该追求的，不是“更像一个炫酷 AI demo”，而是：

> **更像一个稳定的生产平台。**

也就是：

- 调得动
- 看得见
- 解释得清
- 出问题能回退
- 优化后能被量化证明

只要按这个路线走，你这条 AI 视频转 GIF 链路会非常有长期价值。
