# 第三方工具接入路线图（按项目现状拆到模块 / 接口 / 脚本级别）

> 适用仓库：`/Users/mac/go/src/emoji/backend` 当前工作区  
> 目标：在不推翻现有 Go + Asynq + ffmpeg/ffprobe + 七牛 + AI 流水线的前提下，引入高 ROI 的第三方工具，提升 GIF/静态图生产质量、体积控制、可回归性与可观测性。

---

## 1. 一句话结论

按你们当前项目现状，**最值得接入的顺序**是：

1. **Gifsicle**：先解决 GIF 体积与发布优化问题
2. **PySceneDetect**：再提升候选窗口召回质量
3. **VMAF / libvmaf**：补齐离线 benchmark 与参数回归基线
4. **imgproxy**：用于分发层缩略图/封面图派生，不放入主生产链
5. **Real-ESRGAN**：只做可选静态图增强支路，不做默认主链路
6. **Asynqmon + OpenTelemetry**：补齐队列与链路可观测性

我对优先级的判断是：

- **P0 主链路**：`Gifsicle + PySceneDetect + VMAF`
- **P1 分发层**：`imgproxy`
- **P2 可选增强**：`Real-ESRGAN`
- **P0 平台支持**：`Asynqmon + OpenTelemetry`

### 2.2 这意味着什么

这意味着你们**不需要大改架构**，只需要把第三方工具插到下面几个稳定位置：

1. **分析前**：source probe 后、候选生成前
2. **渲染后**：GIF 输出完成后、上传前
3. **离线评估**：任务结束后或 baseline 回归脚本中
4. **分发层**：结果展示与缩略图派生时
5. **平台层**：Worker 队列监控、trace/metric/log 打点

## 3. 工具优先级与适用范围

| 工具 | 建议优先级 | 放置位置 | 主要收益 | 是否进主链路 |

| Asynqmon | P0 | 平台层 | 队列/任务监控 | 否 |
| OpenTelemetry | P0 | 平台层 | trace + metrics + logs | 否 |

## 4. 逐个工具的模块级接入方案

## 4.1 Gifsicle：优先接，收益最快

### 目标

在 `ffmpeg` 已经生成 GIF 之后，再做一次无损/近无损优化，重点降低：

- `size_bytes`
- 包内总体积
- 大 GIF 的失败率/分发压力

### 最佳接入点

优先接在：

- `internal/videojobs/processor_gif_render.go`
- `internal/videojobs/processor_animated_task.go`
- `internal/videojobs/gif_rerender.go`

建议流程：

```text
ffmpeg render -> gifsicle optimize -> stat file -> upload -> persist metadata -> evaluation
```

### 建议新增文件

- `internal/videojobs/tool_runtime.go`
  - 统一探测 `gifsicle` / `ffmpeg` / `ffprobe` / `python3`
- `internal/videojobs/tool_gifsicle.go`
  - `optimizeGIFWithGifsicle(...)`
  - `detectGifsicleAvailable()`
- `internal/videojobs/tool_gifsicle_test.go`

### 建议改动点

#### A. `internal/videojobs/capabilities.go`

把当前仅返回 `ffmpeg/ffprobe` 的能力检测，扩展成：

- `gifsicle_available`
- `pyscenedetect_available`
- `vmaf_available`
- `real_esrgan_available`

#### B. `internal/videojobs/processor_gif_render.go`

在 `renderGIFOutput(...)` 完成后增加：

- 如果 `qualitySettings.GIFGifsicleEnabled == true`
- 且 `gifsicle` 可用
- 则对输出 GIF 跑一次优化

建议 metadata 写入：

```json
{
  "gif_optimization_v1": {
    "tool": "gifsicle",
    "applied": true,
    "before_size_bytes": 1234567,
    "after_size_bytes": 876543,
    "saved_bytes": 358024,
    "saved_ratio": 0.29,
    "profile": "default"
  }
}
```

#### C. `internal/videojobs/gif_evaluation.go`

把 `gif_optimization_v1` 同步进 `feature_json`，后续做后台报表时可以直接看：

- 优化前后体积变化
- 优化是否影响 loop / clarity / overall

### 后台接口级改动

#### `GET /api/video-jobs/capabilities`

扩展响应：

```json
{
  "ffmpeg_available": true,
  "ffprobe_available": true,
  "gifsicle_available": true,
  "external_tools": [...]
}
```

#### `GET/PUT/PATCH /api/admin/video-jobs/quality-settings`

新增可配项：

- `gif_gifsicle_enabled`
- `gif_gifsicle_level`
- `gif_gifsicle_max_size_gain_threshold`
- `gif_gifsicle_skip_below_kb`

### 脚本级改动

建议新增：

- `scripts/check_gif_optimizer_gain.sql`
- `scripts/check_gif_optimizer_gain.sh`
- `scripts/smoke_gifsicle_release_gate.sh`

重点检查：

- 平均节省体积
- top 95 大图的节省比例
- 优化失败率
- 优化后是否触发更多 `need_manual_review`

### 是否建议先落地

**强烈建议。**这是你们当前项目中最容易立刻见收益的一项。

## 4.2 PySceneDetect：第二优先级，提升候选召回质量

### 目标

替代当前仅依赖 `ffmpeg select=gt(scene,...)` 的轻量切镜能力，把候选窗口建立在更稳定的 scene/shot 边界上。

### 当前代码接入点

你们现在已有：

- `inferSceneTags(...)`
- `detectScenePoints(...)`
- `buildFallbackHighlightCandidates(...)`
- `pickNonOverlapCandidates(...)`

这些逻辑主要在：

- `internal/videojobs/processor_scene_selection.go`
- `internal/videojobs/processor_gif_candidates.go`

所以最合理的接入方式是：

```text
probeVideo -> PySceneDetect -> scene list -> candidate generator -> AI2 / scoring / render
```

### 建议新增文件

- `internal/videojobs/tool_scenedetect.go`
  - `runPySceneDetect(...)`
  - `loadSceneDetectJSON(...)`
- `internal/videojobs/tool_scenedetect_test.go`

### 接入策略

#### Phase 1：旁路接入，不替换主逻辑

先保留现有 `detectScenePoints(...)`，新增一条分支：

- 若 `pyscenedetect` 可用并且后台开关打开，则优先使用 PySceneDetect
- 否则回退到现有 ffmpeg scene score

#### Phase 2：scene-aware candidate generation

在 `processor_gif_candidates.go` 里，让候选窗口优先从 scene 边界内构造，而不是只靠时间切片。

建议 metrics 写入：

```json
{
  "scene_detect_v1": {
    "tool": "pyscenedetect",
    "applied": true,
    "detector": "adaptive",
    "scene_count": 14,
    "avg_scene_sec": 3.8,
    "fallback_used": false
  }
}
```

### 后台接口级改动

#### `GET/PUT/PATCH /api/admin/video-jobs/quality-settings`

新增：

- `scene_detect_enabled`
- `scene_detect_provider`（`ffmpeg` / `pyscenedetect`）
- `scene_detect_detector`（`content` / `adaptive`）
- `scene_detect_threshold`
- `scene_detect_min_scene_len_sec`

#### `GET /api/admin/video-jobs/:id/gif-audit-chain`

增加返回：

- `scene_detect_provider`
- `scene_count`
- `scene_detect_fallback`

#### 可选新增接口

- `POST /api/admin/video-jobs/:id/scene-detect-preview`

用于只跑 scene detect，不做正式 rerender，方便后台调参数。

### 脚本级改动

建议新增：

- `scripts/run_scene_detect_baseline_flow.sh`
- `scripts/check_scene_detect_quality.sql`
- `scripts/check_scene_detect_quality.sh`

建议指标：

- candidate 命中率
- top1 proposal 接受率
- rerender 率
- hard gate block 率
- deliver 率变化

### 是否建议先落地

**建议作为第二阶段。**它对“选得准不准”比对“渲染得清不清”更有价值。

## 4.3 VMAF / libvmaf：放离线 benchmark，不进用户主链路

### 目标

给你们现有的 profile、loop tune、rerender、batch compare 提供一个**可重复的客观对照指标**。

### 最佳接入位置

不要放在用户请求主链路；建议放在：

- `scripts/run_video_jobs_baseline_flow.sh`
- `scripts/compare_video_jobs_baseline.py`
- 新增的 benchmark worker/task
- admin rerender 后的异步离线评估

### 为什么不建议进主链路

因为它更适合：

- 对比不同渲染参数组合
- 对比 `ffmpeg only` vs `ffmpeg + gifsicle`
- 对比旧 profile vs 新 profile

而不是每个在线任务都跑一遍。

### 建议新增文件

- `internal/videojobs/tool_vmaf.go`
- `internal/videojobs/tool_vmaf_test.go`
- `internal/videojobs/benchmark_vmaf.go`

### 建议新增 task

在 `internal/videojobs/tasks.go` 增加：

- `video_jobs:benchmark_vmaf`

payload 可以带：

- `job_id`
- `output_id`
- `baseline_output_id`
- `profile_name`
- `run_reason`

### 数据落地建议

#### 第一阶段：先写 JSON

先把结果写进：

- `output.metadata.vmaf_benchmark_v1`
- 或 `job.metrics.vmaf_benchmark_summary_v1`

#### 第二阶段：再上表

如需长期分析，再加：

- `ops.video_tool_benchmarks`
- `audit.video_tool_runs`

### 后台接口级改动

建议新增：

- `POST /api/admin/video-jobs/:id/benchmark-vmaf`
- `GET /api/admin/video-jobs/:id/benchmark-vmaf`

也可以更保守一点：

- 不开放新接口
- 直接挂到已有 `rerender-gif` / baseline flow 脚本之后触发

### 脚本级改动

建议新增：

- `scripts/run_vmaf_regression_flow.sh`
- `scripts/check_vmaf_regression.sql`
- `scripts/check_vmaf_regression.sh`

重点对比：

- `overall_score` / `loop_score` / `size_bytes`
- `vmaf_mean`
- `vmaf_p5`
- `banding / CAMBI`（如果后续引入）

### 是否建议先落地

**建议尽快上，但只上离线链路。**这是你们做“参数上线前回归”的关键基础设施。

## 4.6 Asynqmon + OpenTelemetry：平台层必补

### Asynqmon

你们已经用了 `github.com/hibiken/asynq`，所以加 `Asynqmon` 的成本很低。

建议用途：

- 看队列堆积
- 看失败重试
- 看任务延迟
- 看 worker 消费健康度

### OpenTelemetry

建议覆盖：

- HTTP handler trace
- Worker task trace
- `ffmpeg/ffprobe` 外部命令 span
- AI 调用 span
- 七牛上传/下载 span

### 建议新增文件

- `internal/service/observability.go`
- `internal/middleware/otel_http.go`
- `internal/videojobs/otel_task.go`

### 接口级改动

无需对外 API。

### 是否建议先落地

**建议和 P0 一起做最小版本。**否则后面工具接得越多，越难定位性能/失败问题。

## 8. 分阶段落地计划

## Phase A：1 周内先接 Gifsicle

### 目标

- 不改候选逻辑
- 不改 AI
- 只在 GIF render 后多一步优化

### 成功标准

- 平均 GIF 体积下降 10%~30%
- 发布失败率不上升
- 质量投诉不明显上升

---

## Phase B：1~2 周接 PySceneDetect

### 目标

- 先旁路比对，再切正式流量

### 成功标准

- proposal top1 接受率提升
- rerender 率下降
- deliver 率稳定或上升

---

## Phase C：1~2 周补 VMAF 回归链路

### 目标

- 所有 profile 改动、loop tune 调整、gifsicle 接入，都有离线回归对照

### 成功标准

- baseline flow 能产出 VMAF 对比结果
- 参数上线不再只看人工主观样本

---

## Phase D：分发层接 imgproxy

### 目标

- 把缩略图/封面派生从业务代码里移出去

### 成功标准

- 前台/后台预览图 URL 统一
- 业务层不再自己拼各种裁剪图逻辑

---

## Phase E：可选静态增强支路

### 目标

- 只为运营图/封面图提供加分项，不影响保真主链路
