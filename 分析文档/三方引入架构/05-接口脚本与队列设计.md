# 接口、脚本与队列设计

> 目标：把第一期新增能力对应的 API、脚本、异步任务设计统一下来，避免边写边散。

---

## 1. API 设计原则

1. 用户侧接口尽量不增加复杂度
2. 工具级能力优先放后台与脚本
3. benchmark 与 preview 走异步或 admin 专用入口
4. 所有新增接口都要能表达：是否启用、是否 fallback、执行结果摘要

---

## 3. 新增后台接口建议

## 3.1 `POST /api/admin/video-jobs/:id/benchmark-vmaf`

### 用途

异步触发一个任务或一个 output 的 benchmark。

### 请求体建议

```json
{
  "output_id": 123,
  "baseline_output_id": 456,
  "sample_rate": 10,
  "reason": "profile_compare"
}
```

### 响应体建议

```json
{
  "accepted": true,
  "task_type": "video_jobs:benchmark_vmaf",
  "job_id": 1001,
  "output_id": 123
}
```

---

## 3.2 `POST /api/admin/video-jobs/:id/scene-detect-preview`

### 用途

只跑 scene detect，返回 scene 列表和摘要，不落正式产物。

### 请求体建议

```json
{
  "provider": "pyscenedetect",
  "detector": "adaptive",
  "threshold": 27,
  "min_scene_len_sec": 0.8
}
```

### 响应体建议

```json
{
  "provider": "pyscenedetect",
  "detector": "adaptive",
  "scene_count": 12,
  "avg_scene_sec": 3.2,
  "boundaries": [
    {"start_sec": 0.0, "end_sec": 1.9},
    {"start_sec": 1.9, "end_sec": 4.2}
  ]
}
```

## 4. 异步任务设计

当前已有：

- `video_jobs:process`

第一期建议新增：

## 4.1 `video_jobs:benchmark_vmaf`

### payload

```json
{
  "job_id": 1001,
  "output_id": 123,
  "baseline_output_id": 456,
  "sample_rate": 10,
  "reason": "profile_compare"
}
```

### 处理逻辑

1. 读取 output 与 baseline output
2. 下载素材
3. 跑 VMAF
4. 写 metadata / metrics / audit

---

## 4.2 是否新增 `scene_detect_preview` task？

**第一期不建议。**

scene detect preview 可以先同步跑，因为：

- 只在 admin 工具里用
- 返回快
- 更利于调参

如果后面 preview 越来越重，再异步化。

## 5. 脚本设计

你们现有脚本风格已经很统一，建议完全沿用。

## 5.1 Gifsicle 相关

### `scripts/check_gif_optimizer_gain.sql`

建议输出：

- 样本数
- 优化前平均大小
- 优化后平均大小
- 平均节省比例
- p50/p90/p95 节省比例
- 优化失败数

### `scripts/smoke_gifsicle_release_gate.sh`

建议检查：

- 开启 Gifsicle 后，任务成功率无明显下降
- 大图体积下降显著
- 失败时能 fallback 原图

---

## 5.2 Scene detect 相关

### `scripts/run_scene_detect_baseline_flow.sh`

建议流程：

1. 选定固定视频样本集
2. 运行旧逻辑
3. 运行 PySceneDetect 旁路
4. 输出 scene count / candidate overlap / top1 命中差异

### `scripts/check_scene_detect_quality.sql`

建议输出：

- 平均 candidate 数
- average proposal accept rate
- rerender 率
- deliver 率
- manual review 率

---

## 5.3 VMAF 相关

### `scripts/run_vmaf_regression_flow.sh`

建议流程：

1. 跑 baseline outputs
2. 跑 candidate / rerender outputs
3. 调 benchmark-vmaf
4. 聚合结果

### `scripts/check_vmaf_regression.sql`

建议输出：

- `vmaf_mean`
- `vmaf_p5`
- `overall_score`
- `loop_score`
- `size_bytes`
- `saved_ratio`

---

## 7. 返回语义建议

所有新增接口和脚本输出，尽量统一 3 个语义字段：

- `applied`
- `fallback_used`
- `error`

这样后台、脚本、任务、审计链可以统一消费。
