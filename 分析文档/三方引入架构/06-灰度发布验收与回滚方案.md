# 灰度发布、验收与回滚方案

> 目标：确保三方工具接入后，不会因为新工具的偶发失败、二进制缺失、参数不稳而直接伤到主链路。

---

## 1. 灰度原则

1. **所有新能力都要有开关**
2. **所有新工具都要有 fallback**
3. **先旁路、后主用、再扩大流量**
4. **先 admin / baseline，再用户生产**

---

## 2. Gifsicle 灰度方案

## Phase 1

- 在测试环境开启
- 只跑 baseline 与 rerender 路径
- 检查体积下降与失败率

## Phase 2

- 线上开启但仅针对：
  - `gif` 主格式
  - 大于阈值的 GIF
- 先全量记录，不强依赖

## Phase 3

- 对正式生产全量生效
- 保持失败自动 fallback 原始 GIF

### 验收指标

- 平均 `size_bytes` 下降 ≥ 10%
- `job done rate` 不下降超过 0.5%
- `manual review` 不显著恶化

### 回滚方式

- 后台开关关闭：`gif_gifsicle_enabled=false`
- 保留代码，不影响主 render

---

## 3. PySceneDetect 灰度方案

## Phase 1：旁路观测

- 跑 scene detect，但不影响 candidate 最终结果
- 只记录 metrics

### 观察指标

- scene count 稳定性
- fallback rate
- scene detect failure rate

## Phase 2：小流量消费

- 只让 admin rerender / baseline flow 消费 scene list

## Phase 3：正式候选链路接入

- 先 10%
- 再 30%
- 再 100%

### 验收指标

- top1 proposal accept rate 上升
- rerender rate 下降
- hard gate blocked rate 不恶化

### 回滚方式

- `scene_detect_enabled=false`
- 或 provider 切回 `ffmpeg`

---

## 4. VMAF 灰度方案

VMAF 第一阶段不进用户主链路，所以灰度重点不是“流量”，而是“范围”。

## Phase 1

- 仅本地 / CI / baseline flow

## Phase 2

- admin benchmark-vmaf 手动触发

## Phase 3

- rerender 后自动异步触发（可选）

### 验收指标

- benchmark 成功率高
- 样本输出稳定
- 结果和主观判断基本一致

### 回滚方式

- 直接关闭 `vmaf_benchmark_enabled`
- 停止投递 `video_jobs:benchmark_vmaf`

---

## 5. 平台可观测性验收

## Asynqmon

至少能看到：

- 队列积压
- retry 数
- dead/archived 任务
- benchmark-vmaf 任务耗时

## OpenTelemetry

至少能追到：

- 一个 job 从 handler 到 task 再到 ffmpeg/gifsicle 的链路
- 外部命令失败时有 span error
- 上传/下载耗时可见

## 7. 回滚等级设计

## Level 1：关闭后台开关

适用于：

- 参数不稳定
- 收益不明显
- 轻度失败

## Level 2：切回旧 provider

适用于：

- PySceneDetect 效果不稳定

处理方式：

- `scene_detect_provider=ffmpeg`
- `scene_detect_enabled=false`

## Level 3：停异步任务

适用于：

- benchmark 队列异常

处理方式：

- 不投递 `video_jobs:benchmark_vmaf`
- 保留主生产流不受影响

## Level 4：回滚二进制依赖

适用于：

- 工具本身版本问题
- worker 环境污染

处理方式：

- 回滚镜像 / worker runtime
- Config 指向旧二进制或关闭功能

---

## 8. 第一阶段验收门槛（建议）

## Gifsicle

- 成功率 ≥ 99%
- 失败时 fallback 成功率 = 100%
- 平均体积下降 ≥ 10%

## PySceneDetect

- fallback rate ≤ 5%
- top1 accept 不下降
- rerender rate 不上升超过 3%

## VMAF

- benchmark 成功率 ≥ 95%
- baseline 差异结果可复现

## 平台

- capabilities 响应正确
- audit chain 可见新字段
- worker trace 可串联外部命令

---

## 9. 最终建议

第一期上线顺序建议固定成：

1. **先上 Gifsicle**
2. **再上 PySceneDetect 旁路**
3. **再补 VMAF benchmark**
4. **最后补 Asynqmon / OTel 完善观测**

不要反过来从最重的、最不确定的生成式增强开始。
