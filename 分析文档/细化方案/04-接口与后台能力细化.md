# 接口与后台能力细化

## 1. 目标

当前仓库已经有：

- 用户任务入口
- Prompt 模板后台
- GIF 健康看板
- proposal 重渲后台

细化方案的目标不是重写 API，而是把现有能力补成一套完整的：

> **创建任务、查看链路、调 Prompt、看健康、做人审、做重渲、做回放**

的后台工具集。

---

## 2. 当前已存在的关键接口

## 2.1 用户任务入口

### 已存在

- `POST /api/video-jobs`

### 对应代码

- `internal/handlers/video_jobs.go`

### 作用

- 创建视频转图任务
- 校验与入队

---

## 2.2 Prompt 模板后台

### 已存在路由

- `GET /api/admin/video-jobs/ai-prompt-templates`
- `PATCH /api/admin/video-jobs/ai-prompt-templates`
- `GET /api/admin/video-jobs/ai-prompt-templates/audits`
- `GET /api/admin/video-jobs/ai-prompt-templates/versions`
- `POST /api/admin/video-jobs/ai-prompt-templates/activate`

### 对应代码

- `internal/handlers/video_ai_prompt_templates.go`
- `internal/router/router.go`

### 现有能力评价

这已经是生产级 prompt 治理接口雏形，建议继续保留并增强，而不是重做。

---

## 2.3 GIF 健康看板

### 已存在路由

- `GET /api/admin/video-jobs/gif-health`
- `GET /api/admin/video-jobs/gif-health/trend`
- `GET /api/admin/video-jobs/gif-health/trend.csv`

### 对应代码

- `internal/handlers/admin_video_jobs_gif_health.go`

### 当前价值

已可用于：

- SQL schema 健康检查
- jobs/outputs 总览
- loop tune 统计
- candidate reject 概览
- 失败原因汇总

这是非常重要的现成资产。

---

## 2.4 proposal 重渲接口

### 已存在路由

- `POST /api/admin/video-jobs/:id/rerender-gif`

### 对应代码

- `internal/handlers/admin_video_jobs_rerender.go`
- `internal/videojobs/gif_rerender.go`

### 当前能力

- 支持 `proposal_id` 或 `proposal_rank`
- 支持 `force`
- 产出 output / emoji / artifact / cost delta
- 会记录 audit log

这已经足够作为运营补偿工具的底座。

## 3. 建议补齐的后台能力

## 3.1 建议新增：任务链审计接口

### 建议路由

- `GET /api/admin/video-jobs/:id/gif-audit-chain`

### 作用

一次返回：

- job 基本信息
- candidate 列表
- directive
- proposal 列表
- outputs
- evaluations
- reviews
- feedback
- rerender 记录

### 为什么必须要有

因为调优时最重要的问题不是“整体平均分是多少”，而是：

> **这个 case 为什么这样走？**

没有一条完整 audit chain，线上调优会很痛苦。

---

## 3.2 建议新增：人工审核决策接口

### 建议路由

- `POST /api/admin/video-jobs/:id/gif-review-decisions`

### 请求示例

```json
{
  "items": [
    {
      "output_id": 123,
      "decision": "deliver",
      "reason": "semantic_good_but_boundary_technical_score",
      "notes": "运营确认可交付"
    }
  ]
}
```

### 作用

- 给 `need_manual_review` 的样本打最终标签
- 写 `ops.video_job_gif_manual_scores` 或单独审计日志
- 允许运营覆盖“是否上架”层面的业务状态

### 注意

人工审核可以覆盖**运营交付状态**，但不要反向改写技术事实分。

---

## 3.3 建议新增：批量重渲接口

### 建议路由

- `POST /api/admin/video-jobs/:id/rerender-gif/batch`

### 请求示例

```json
{
  "proposal_ids": [101, 102, 103],
  "strategy": "loop_first",
  "force": false
}
```

### 作用

- 对多个 proposal 做批量试渲
- 用于运营挑选与模型回放

### 建议策略枚举

- `default`
- `loop_first`
- `size_first`
- `clarity_first`
- `viral_first`

---

## 3.4 建议新增：Prompt 实验/灰度查看接口

### 建议路由

- `GET /api/admin/video-jobs/ai-experiments`
- `GET /api/admin/video-jobs/ai-experiments/:id`

### 作用

- 看某个 prompt 版本的样本量
- 看各版本 deliver rate / cost / manual review rate
- 看版本切换后的趋势变化

### 当前实现建议

如果暂时没有 experiment 表，可先聚合：

- `ops.video_ai_prompt_templates`
- `ops.video_job_ai_usage`
- `archive.video_job_gif_*`
- `public.video_image_feedback`

---

## 4. 后台页面建议

## 4.1 Job Detail 页面

### 应展示

- 任务基础信息
- 视频 meta
- quality settings snapshot
- AI sub-stage 状态
- candidates
- directive
- proposals
- outputs
- evaluations
- reviews
- feedback
- rerender history

### 推荐布局

```text
基本信息
  -> 输入视频
  -> 质量与渲染设置
  -> AI1 brief
  -> AI2 proposals
  -> 输出结果
  -> 技术评分
  -> AI3 review
  -> hard gate
  -> 用户反馈
  -> rerender 操作区
```

---

## 4.2 Prompt Template 页面

### 现有就应支持

- 按 format/stage/layer 查看
- 查看 active 版本
- 修改模板正文
- 激活指定版本
- 查看审计日志

### 建议增强

- 关联“最近使用样本数”
- 关联“最近 7 天效果摘要”
- 支持只读对比两个版本 diff

---

## 4.3 GIF Health 页面

### 当前已有统计基础

- jobs 总览
- outputs 总览
- loop tune
- consistency
- integrity
- candidate reject

### 建议增强两个 tab

#### A. Delivery Tab

展示：

- deliver rate
- internal rate
- reject rate
- manual review rate
- judge/hard gate 冲突率

#### B. Cost Tab

展示：

- avg ai cost
- avg total cost
- 单 deliver 样本成本
- 各 stage tokens 与耗时

## 5. API 契约建议

## 5.1 所有后台接口建议带版本视图信息

例如：

```json
{
  "policy_version": "gif_pipeline_policy_v2",
  "prompt_version": "gif_planner_v3",
  "selection_version": "gif_render_selection_v1"
}
```

### 原因

方便前端与运营理解“当前看到的是哪一版逻辑”。

---

## 5.2 所有 AI 相关响应建议回传 resolved 信息

如：

- `resolved_from=gif|all`
- `template_version`
- `is_active`
- `enabled`

当前 Prompt Template 接口已经开始这样做，建议推广到更多审计接口。

---

## 5.3 所有人审/重渲接口建议带审计日志

### 要记录的字段

- actor_id
- actor_role
- reason
- before_status
- after_status
- output_id / proposal_id
- cost delta
- trigger

### 原因

避免后期出现“到底是谁做了这次人工干预”无法追溯的问题。

---

## 6. 用户侧接口建议

用户侧原则：

- 简洁
- 稳定
- 不暴露内部 AI 复杂度

### 用户侧只需要关心

- 任务是否创建成功
- 当前进度
- 结果是否可下载
- 哪些结果可交付

### 不建议直接暴露给用户的内部概念

- directive
- raw proposal
- judge 原始 reason
- hard gate 内部阈值

这些更适合后台。

---

## 7. 接口优先级建议

如果按投入产出比排序，我建议优先做：

1. `GET /api/admin/video-jobs/:id/gif-audit-chain`
2. `POST /api/admin/video-jobs/:id/gif-review-decisions`
3. `POST /api/admin/video-jobs/:id/rerender-gif/batch`
4. `GET /api/admin/video-jobs/ai-experiments`

原因：

- 先把 case 调优链路打通
- 再补人工闭环
- 再补批量运营能力
- 最后再做更完整的实验看板

---

## 8. 最终建议

从 API 角度，当前系统最缺的不是“更多接口”，而是：

- 一条完整审计接口
- 一条人工定结论接口
- 一条批量重渲接口

只要这三条补起来，你的 AI GIF 系统就会从“功能可用”提升到“运营可调、问题可查、效果可追”。
