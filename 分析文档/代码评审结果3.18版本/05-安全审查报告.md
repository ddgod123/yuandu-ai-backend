# 05-安全审查报告

> 审查日期：2026-03-19  
> 审查范围：当前工作区代码（不含线上部署配置）

## 1. 结论摘要

当前后端在**短信限流、验证码、风控事件、黑名单、下载票据**等方面做得不错，但存在若干**非常高风险**的问题，尤其是：

1. **公共路由中暴露了多组写接口**
2. **公共路由中暴露了对象存储管理能力**
3. **示例环境文件中疑似包含真实第三方密钥**

如果这是线上同构代码，这几项都应当按**最高优先级**处理。

---

## 2. 风险评级概览

| 级别 | 数量 | 结论 |
|---|---:|---|
| Critical | 2 | 需要立即修复 |
| High | 3 | 本周内修复 |
| Medium | 4 | 进入近期迭代 |
| Low | 3 | 可顺手治理 |

---

## 3. 已有的正向安全能力

这些部分是当前系统的加分项：

- 图形验证码：`auth_security.go`
- 短信发送限流：`sms_limiter.go`
- 登录失败锁定：`markAuthVerifyFailure`
- 风险事件记录：`recordRiskEvent`
- 自动拉黑：`maybeAutoBlacklistFromRiskEvent`
- 下载票据：IP / UA / 用户绑定
- Redis 作为验证码与票据的一次性消费存储
- 后台路由有 `RequireAnyRole("super_admin")`

这说明你的风控意识是在线的，问题主要集中在**路由暴露面与配置管理**。

---

## 4. Critical 级问题

## C1. 公共内容写接口未受保护

### 现象

这些接口位于公共 `/api` 分组，只挂了 `AuthOptional`：

- `POST /api/collections`
- `PUT /api/collections/:id`
- `DELETE /api/collections/:id`
- `POST /api/emojis`
- `PUT /api/emojis/:id`
- `DELETE /api/emojis/:id`

对应 handler 中没有看到强制登录、所有者校验或管理员校验。

### 风险

匿名调用者可能直接：

- 创建合集
- 修改合集
- 删除合集
- 批量上传表情
- 修改/删除表情

这是**内容面完全可写**的风险。

### 代码位置

- 路由：`internal/router/router.go`
- Handler：`internal/handlers/collections.go`
- Handler：`internal/handlers/emojis.go`

### 建议

立即处理：

1. 将这些接口移动到认证组或管理员组
2. 对普通用户接口增加 owner 校验
3. 对管理接口只允许 admin/super_admin
4. 给删除操作补审计日志

---

## C2. 公共存储管理接口暴露对象写能力

### 现象

以下接口当前也是公共组：

- `POST /api/storage/upload-token`
- `DELETE /api/storage/object`
- `POST /api/storage/rename`
- `GET /api/storage/objects`
- `GET /api/storage/object`
- `GET /api/storage/url`
- `POST /api/storage/urls`

### 风险

匿名调用者可能：

- 申请七牛上传 token
- 删除对象
- 重命名对象
- 枚举对象列表
- 获取对象详情与 URL

尤其 `GetUploadToken` 允许生成 `emoji/` 前缀下 token，这本质上是把对象写权限暴露给了未鉴权调用者。

### 代码位置

- 路由：`internal/router/router.go`
- 逻辑：`internal/handlers/storage.go`

### 建议

立即处理：

1. `upload-token` 至少要求登录
2. 删除 / rename / list / object info 改到 admin 组
3. 如果用户侧确实需要上传 token，按用户/collection 精细发放 prefix，不允许任意 `emoji/` 根前缀
4. URL 签名接口至少加鉴权和审计

## 5. High 级问题

## H1. `.env.example` 中疑似存在真实第三方密钥

### 现象

示例文件中出现了看起来像真实的：

- `AI_PLANNER_API_KEY=...`
- `AI_JUDGE_API_KEY=...`

### 风险

- 泄露第三方模型额度
- 被他人盗刷
- 一旦仓库共享/备份/镜像外传，风险很大

### 建议

1. 立即轮换相关 key
2. 将 `.env.example` 只保留空值或占位值
3. 如有必要，清理 Git 历史

---

## H2. JWT 默认密钥过弱

### 现象

`config.Load()` 中默认：

- `JWT_SECRET = change-me`

### 风险

如果部署环境忘记覆盖，就可能出现可预测签名密钥。

### 建议

- 在 `prod` 环境下强制要求设置非默认值
- 启动时校验，默认值直接拒绝启动

---

## H3. 管理员一键拉起 Worker 本质上会执行 shell 命令

### 现象

`/api/admin/system/worker-start` 会执行：

- `WORKER_START_COMMAND`
- 通过 `/bin/sh -lc` 调用

### 风险

虽然它在 admin 路由下，但本质上仍是：

- 远程执行 shell
- 依赖环境变量是否安全配置

如果后台账号或配置被拿下，危害很大。

### 建议

- 只允许白名单命令
- 或者改成固定 supervisor/systemd hook
- 保留操作审计

---

## 6. Medium 级问题

## M1. JWT 解析未显式校验签名算法

### 现象

`jwt.ParseWithClaims` 时仅返回 secret，没有显式检查 `token.Method` 是否符合预期。

### 风险

依赖库默认行为虽可能已足够，但从防御性编码角度，仍建议显式限制算法。

### 建议

- 校验 `SigningMethodHMAC` 且预期 `HS256/HS512` 等固定算法

---

## M2. 公共对象代理接口仍然扩大了暴露面

### 现象

`/api/storage/proxy` 虽然限制为 `emoji/` 前缀，但仍可匿名代理对象内容。

### 风险

- 带宽放大
- 扫描/滥刷资源
- 绕开原本更细粒度的下载控制

### 建议

- 对代理接口做签名校验或票据校验
- 至少增加速率限制和审计

---

## M3. 公共列表与对象探查接口利于资产枚举

### 现象

- `GET /api/storage/objects`
- `GET /api/storage/object`

### 风险

会降低对象命名空间的隐藏性，便于爬取或探测存储结构。

### 建议

- 改 admin-only
- 或至少要求登录并限制 prefix

---

## M4. 部分“用户写操作”缺少统一审计

### 现象

内容删除、对象删除、重命名等操作，没有统一审计覆盖。

### 风险

事后溯源困难。

### 建议

- 对 destructive 操作统一补 `audit_logs`

---

## 7. Low 级问题

## L1. `MINIO_*` 配置残留但未真正投入使用

风险不大，但容易造成误解。建议要么接入，要么删掉。

## L2. 仓库根目录存在编译产物 `api` / `worker`

不应长期进入代码仓库，建议移出并补 `.gitignore`。

## L3. README 几乎为空

从安全角度看，不利于建立正确的启动和配置基线。这个任务里已顺手补文档。

---

## 8. 风险优先级修复顺序

### 第 1 优先级（立即）

1. 关闭公共内容写接口
2. 关闭公共存储管理接口
3. 轮换并移除示例环境文件中的真实 key

### 第 2 优先级（本周）

4. 强制生产环境自定义 JWT secret
5. 收紧 worker-start 的执行模型
6. 给破坏性操作补审计

### 第 3 优先级（近期）

7. JWT 显式校验算法
8. 收紧 proxy / signed URL / object inspect 接口
9. 清理配置残留与仓库产物

---

## 9. 建议的最小修复方案

如果你只想快速止血，我建议马上做这几条：

1. 把以下路由迁到认证/管理员组：
   - `/api/collections` 的 POST/PUT/DELETE
   - `/api/emojis` 的 POST/PUT/DELETE
   - `/api/storage/*` 除必要只读下载外的大部分接口
2. 将 `.env.example` 中所有 key 清空
3. 在 `prod` 启动时阻止默认 JWT secret
4. 给对象删除/重命名/内容删除增加审计日志

---

## 10. 最终结论

当前系统的**业务风控思维是好的**，但**路由暴露面**上存在严重反差：

> 内部做了很多验证码、黑名单、下载票据、防刷；但外部却仍暴露了不该公开的写接口和对象管理接口。

这类问题一旦在线上存在，优先级应当高于大多数功能迭代。
