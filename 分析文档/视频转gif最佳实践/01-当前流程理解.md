# 当前视频转 GIF 流程理解

> 本文基于当前工作区代码，不是抽象想象。

## 1. 总体判断

我对你当前方案的理解是：

> **规则引擎负责确定性边界，AI 负责语义增强与策略判断，最终结果仍由技术硬门槛和业务规则兜底。**

也就是说，你现在的 AI 更像：

- `Director`：定义这次任务应该产出什么样的 GIF
- `Planner`：在已有候选基础上做语义提名和排序
- `Judge`：对最终生成产物做语义复审

而不是让 AI 直接替代：

- 视频分析全流程
- 渲染执行
- 技术质量判定
- 最终交付门槛

这是一条更稳的生产系统路线。

---

## 2. 任务入口：CreateVideoJob

关键入口：`internal/handlers/video_jobs.go` -> `CreateVideoJob`

用户创建任务时，后端先完成一轮**确定性前置校验**：

1. 校验用户身份
2. 解析并标准化 `source_video_key`
3. 校验请求参数（标题、格式、编辑参数等）
4. 限制任务格式与能力边界
5. 检查 `ffmpeg / ffprobe`
6. 预探测源视频元信息
7. 给出质量模板建议与积分估算
8. 写入 `archive.video_jobs`
9. 同步镜像到 `public.video_image_jobs`
10. 冻结积分
11. 投递 Asynq 异步任务

### 这里体现出的设计取向

你没有把 AI 放在 API 入口的第一层，而是优先让系统自己完成：

- 资源可用性判断
- 参数合法性判断
- 成本预估
- 幂等/账务处理

这是对的。入口层越稳定，后面 AI 波动对业务面的影响越小。

---

## 3. Worker 主处理链

核心执行器：`internal/videojobs/processor.go`  
分阶段编排：`internal/videojobs/processor_pipeline.go`

整体主流程可以概括为：

1. 读取任务并抢占执行权
2. 下载源视频
3. `ffprobe` 分析视频元信息
4. 自动裁黑边 / 裁信箱区（auto crop / auto letterbox）
5. 本地高光窗口选择（local highlight scorer）
6. 进入 GIF AI 子流程
7. 抽帧与质量筛选
8. GIF/静态图渲染
9. 上传七牛
10. 落库 `collection / emoji / artifact`
11. 生成 zip
12. public mirror 同步
13. 成本核算
14. 积分结算
15. 清理源视频与临时文件

### 关键认知

这不是“AI 生成 GIF 系统”，而是：

> **传统视频处理流水线 + AI 决策增强流水线**

真正执行视频切片、抽帧、渲染、上传、入库、结算的，仍然是你自己的后端逻辑。

---

## 4. AI 子流程拆解

当前 GIF 子流程在代码里已经显式拆成了四个子阶段状态：

- `briefing`
- `planning`
- `scoring`
- `reviewing`

其中 AI 实际核心是三个角色。

---

## 5. AI1：Director（任务级策略层）

关键代码：`internal/videojobs/ai_gif_pipeline.go` -> `requestAIGIFPromptDirective`

### 5.1 它做的不是“直接选片”

Director 的职责不是决定第几秒到第几秒，而是先产出一份**任务级策略指令包**。典型字段包括：

- `business_goal`
- `audience`
- `must_capture`
- `avoid`
- `clip_count_min/max`
- `duration_pref_min/max`
- `loop_preference`
- `style_direction`
- `risk_flags`
- `quality_weights`
- `directive_text`

### 5.2 它解决的问题

它本质上是在回答：

- 这次 GIF 产物更偏娱乐传播，还是设计素材？
- 应该偏反应类、动作类，还是节奏类？
- 更看重语义价值、清晰度、循环感，还是体积效率？

### 5.3 Prompt 治理已经具备基础能力

当前实现中，Director 的 prompt 来源不是写死单点，而是支持：

- 内建默认 prompt
- `ops.video_ai_prompt_templates` 中按 `format/stage/layer` 读取模板
- `editable + fixed` 双层结构

这说明你已经开始做：

- prompt 版本化
- 运营可调
- 线上回滚
- 审计复盘

### 5.4 失败时的回退策略

如果 AI 调用失败或 JSON 解析失败，会：

- 生成 fallback directive
- 持久化到 `archive.video_job_gif_ai_directives`
- 记录 AI usage / fallback 状态

这也是正确做法：**AI 失败不应该拖垮主链路。**

---

## 6. AI2：Planner（候选语义提名层）

关键代码：`internal/videojobs/ai_gif_pipeline.go` -> `requestAIGIFPlannerSuggestion`

### 6.1 输入是什么

Planner 不是面对整段原始视频直接开选，而是基于：

- 本地 highlight scorer 的候选窗口
- Director 的策略指令
- 视频 meta
- topN 目标

### 6.2 输出是什么

Planner 输出的是 proposal 列表，典型字段包括：

- `proposal_rank`
- `start_sec / end_sec`
- `score`
- `proposal_reason`
- `semantic_tags`
- `expected_value_level`
- `standalone_confidence`
- `loop_friendliness_hint`

### 6.3 后处理不是交给 AI 自己决定

Planner 输出完后，系统还会继续做：

- proposal 归一化
- 去重叠 / 去冲突
- confidence threshold 过滤
- topN 截断
- proposal rank -> proposal id 映射
- 回填 candidate / proposal 绑定关系

也就是说：

> **AI 负责提名，系统负责约束。**

这是生产可控的关键。

---

## 7. 反馈 rerank：不是一次性判定，而是闭环修正

在 `processor_pipeline.go` 里，Planner 后还有一层基于历史反馈的 rerank。

系统会结合：

- 用户历史偏好
- public feedback 信号
- 负反馈保护（negative guard）

对候选进一步重排，并写入：

- `ops.video_job_gif_rerank_logs`

这个点很重要，因为它说明你已经从“单次任务选片”走向“长期用户偏好学习”。

---

## 8. scoring / rendering：本地质量链仍然是主角

虽然 AI 决定了很多“语义层”问题，但真正决定 GIF 能否落地的仍然是本地链路：

- 候选窗口持久化：`archive.video_job_gif_candidates`
- 结果级评测：`archive.video_job_gif_evaluations`
- 抽帧质量筛选
- GIF 渲染与 loop tuning
- 输出文件上传与入库

所以你当前架构里，AI 主要影响：

- 哪些片段更值得试
- 哪些输出更适合交付

但它没有替代：

- 渲染器
- 质量测量器
- 文件体积控制器
- 工程级失败恢复逻辑

---

## 9. AI3：Judge（结果复审层）

关键代码：`internal/videojobs/ai_gif_pipeline.go` -> `runAIGIFJudgeReview`

### 9.1 输入不是候选，而是成品输出

Judge 阶段读取的是：

- 已生成的 `public.video_image_outputs`
- 对应 `archive.video_job_gif_evaluations`
- output metadata 中的窗口信息 / proposal rank

### 9.2 输出是最终建议，而不是直接写死交付状态

它给出的建议包括：

- `deliver`
- `keep_internal`
- `reject`
- `need_manual_review`

同时输出：

- `semantic_verdict`
- `diagnostic_reason`
- `suggested_action`

### 9.3 最关键的一层：技术硬门槛 Hard Gate

即使 AI Judge 说某个 GIF 可以 `deliver`，系统依然会再过一层技术硬门槛：

- 总分是否达标
- clarity 是否达标
- loop 是否达标
- output_score 是否达标
- 时长是否过短
- 大小是否严重超预算
- evaluation 是否缺失

如果不满足，会被系统强制降级为：

- `reject`
- 或 `need_manual_review`

这是当前设计里最成熟的一点之一：

> **AI 不能越权覆盖技术失败。**

## 12. 我对你当前方案的归纳

如果压缩成一句话：

> **local scorer 先做候选，AI1 做任务策略，AI2 做语义提名，本地规则与反馈再筛，AI3 对成品复审，最终交付仍由技术硬门槛兜底。**

也可以换一种更产品化的话说：

> **你现在的 AI 是“决策层”，不是“渲染层”；是“增强器”，不是“唯一裁判”。**

## 13. 这套设计当前的优点

### 13.1 优点一：职责分层比较清楚

- 本地逻辑负责确定性判断
- AI 负责模糊语义判断
- 技术门槛负责兜底

### 13.2 优点二：不是单点黑盒

你现在已经有：

- fallback
- prompt versioning
- prompt audit
- proposal / review 持久化
- hard gate
- feedback rerank
- rerender

这比“把视频交给大模型，返回几个时间戳”成熟很多。

### 13.3 优点三：数据结构适合后续迭代

因为 proposal、candidate、evaluation、review 都落库了，所以未来无论你想做：

- prompt 优化
- 小模型替代某个阶段
- 人工标注集建设
- 线上回归对比

都已经有较好的数据基础。

## 14. 当前可能的隐患

### 14.1 隐患一：AI 价值仍然高度依赖本地候选质量

如果 local highlight scorer 的候选覆盖不到真正高价值窗口，AI2 也只能在有限集合里做“相对更优”，无法凭空补回所有漏召。

### 14.2 隐患二：AI1/AI2/AI3 之间还可能出现目标漂移

例如：

- Director 偏传播性
- Planner 偏情绪峰值
- Judge 偏技术稳定性

如果三阶段目标函数没有统一，很容易产生“前面选、后面打回”的链路抖动。

### 14.3 隐患三：成本会随链路复杂度上升

当每个 GIF 任务都经历：

- Director
- Planner
- Judge
- rerender
- 人工审核

如果没有分层触发策略，成本很容易被拖高。

### 14.4 隐患四：缺少统一线上质量北极星时，系统容易变成“局部优化”

例如你可能把：

- loop 分数做高了
- 大小做小了
- AI semantic verdict 做高了

但最后用户不下载、不收藏、不传播，说明真实业务价值并没有提升。

---

## 15. 结论

我认为你当前的方向是对的，而且已经具备生产系统气质。

最重要的不是把它改成“更像 AI 产品”，而是继续沿着下面这条路线增强：

> **让 AI 做更擅长的模糊判断，让系统做更擅长的确定性约束；同时把线上效果、成本、反馈和人工纠偏串成闭环。**
