# 全量 HQ、不减张数前提下的 GIF 提速路线图

> 文档日期：2026-03-20  
> 适用范围：`/Users/mac/go/src/emoji/backend` 当前后端实现

## 1. 目标与硬约束

### 1.1 目标
- 全量识别视频中所有高价值表情窗口（高召回，不漏关键高光）。
- 保持 HQ 产出定位（不做“全局降档”策略）。
- 不限制最终 GIF 输出张数（结果交给用户主观选择）。
- 在以上前提下，显著缩短渲染阶段耗时与任务总耗时。

### 1.2 硬约束
- 不采用“减张数换速度”。
- 不采用“把 default/short/long 全局从 HQ 改低档”作为核心方案。
- 不用“牺牲内容完整性”的手段换性能。

---

## 2. 当前瓶颈复盘（基于现网代码与库内数据）

### 2.1 渲染是主瓶颈
近期已完成 GIF 任务中，`rendering -> uploading` 时段显著占比过高；多条任务 render stage 达数百秒。

### 2.2 当前是“多窗口独立重渲”
核心路径：
- `internal/videojobs/processor_gif_render.go`
- `renderGIFOutput(...)`

每个窗口基本独立进行：seek/decode/scale/fps/palettegen/paletteuse，重复计算高。

### 2.3 并发控制过粗
当前 cap 逻辑在：
- `internal/videojobs/processor_animated_task.go` `resolveGIFRenderWorkerCap(...)`
- `internal/videojobs/processor_persist_outputs.go`

主要按“源视频长边/总时长”限流，未按“窗口真实成本”调度，导致墙钟时长被放大。

### 2.4 估算器低估导致执行层压力失真
估算函数：
- `internal/videojobs/processor_gif_candidates.go` `estimateGIFCandidateSizeKB(...)`

当前估算值与真实值偏差大，影响渲染预算、调度和资源策略判断。

---

## 3. 总体策略（不减量、不降HQ）

### 3.1 核心原则
- 识别层继续高召回。
- 渲染层做“复用”而不是“减产”。
- 调度层做“成本感知并发”。
- HQ 定义从“固定重参数”升级为“观感等质 HQ”。

### 3.2 一句话方案
同样全量、同样 HQ、同样不减张数，改执行路径：**少重复算 + 更聪明并行 + 中间结果复用**。

---

## 4. 分阶段路线图

## Phase 0：建立真实成本画像（P0）
目标：先看清成本，再做提速闭环。

### 4.0.1 数据字段扩展（输出级）
在 `video_image_outputs.metadata` 增加埋点：
- `predicted_size_kb`
- `predicted_render_sec`
- `render_cost_units`
- `decode_stage_ms`
- `normalize_stage_ms`
- `palette_stage_ms`
- `gifsicle_stage_ms`
- `loop_tune_stage_ms`
- `bundle_id`
- `cache_hit`

### 4.0.2 估算器升级
改造：`estimateGIFCandidateSizeKB(...)`
- 输入从静态 `meta.Width/Height + default fps/colors` 升级为：
  - 自适应宽度/帧率/色数/时长
  - 历史回归参数（可选）
- 输出从单一 `size_kb` 升级为 `size + render_time` 双预测。

交付标准：能按候选窗口维度稳定预测“体积/耗时/成本单元”。

---

## Phase 1：渲染复用（P0/P1）
目标：保持全量输出，减少重复编码路径。

### 4.1.1 Window Bundling（同视频多窗口分组）
新增分组器：按时间邻近 + 参数相近组合窗口。

建议新增文件：
- `internal/videojobs/processor_gif_bundle_plan.go`

输出：
- `bundle_id`
- `bundle_start/end`
- `windows[]`
- `estimated_bundle_cost`

### 4.1.2 共享中间片段（Mezzanine）
在 bundle 维度先做一次归一化中间产物，再派生多个 GIF。

建议新增文件：
- `internal/videojobs/processor_gif_mezzanine.go`

收益：减少重复 seek/decode/scale/fps。

### 4.1.3 Bundle Render 执行器
新增执行层，按 bundle 渲染而非按窗口逐条渲染。

建议新增文件：
- `internal/videojobs/processor_gif_render_bundle.go`

交付标准：同一 job 内多窗口平均 `render_elapsed_ms` 显著下降。

---

## Phase 2：成本感知调度（P0/P1）
目标：不降数量，缩短总墙钟时间。

### 4.2.1 引入 weighted scheduler
新增：`gif_render_cost_units` 调度模型。

建议新增文件：
- `internal/videojobs/processor_gif_scheduler.go`

逻辑：
- 大窗口高 cost，低并发。
- 小窗口低 cost，可穿插并发。
- 以“总成本令牌桶”而非固定 worker 数限制。

### 4.2.2 解耦 render 并发与 upload 并发
将 `UploadConcurrency` 与 GIF render 并发拆开配置。

建议扩展配置：
- `gif_render_workers`
- `gif_render_max_cost_units`
- `gif_render_bundle_workers`

涉及文件：
- `internal/videojobs/quality_settings.go`
- `internal/videojobs/processor_persist_outputs.go`

---

## Phase 3：分析与 loop tune 结果复用（P1）
目标：减少重复小流程成本。

### 4.3.1 loop tune 样本缓存
当前 `optimizeGIFLoopWindow(...)` 会二次抽帧与分析，可复用前置分析结果。

建议新增文件：
- `internal/videojobs/processor_gif_loop_cache.go`

复用对象：
- motion/quality sample
- loop closure 基础特征
- 已抽样帧索引

### 4.3.2 统一分析缓存上下文
在 job 级建立 `RenderContextCache`，供 AI1/AI2/score/render 共享必要特征。

交付标准：loop tune 与 quality 分析总耗时下降且结果稳定性不下降。

---

## Phase 4：HQ 等质优化（P1/P2）
目标：不降 HQ，只避免“无效重参数 HQ”。

### 4.4.1 HQ 子档（仍然都属于 HQ）
按内容自动匹配 HQ 子型：
- `hq_portrait_reaction`
- `hq_action_burst`
- `hq_caption_meme`
- `hq_flat_background`

涉及文件：
- `internal/videojobs/processor_animated_profile.go`
- `internal/videojobs/ai_gif_pipeline.go`

### 4.4.2 观感优先的参数策略
- 保证观感一致，不追求机械参数上限。
- 通过内容类型分配 fps/colors/width 预算。

交付标准：人工主观评估“HQ 感受不下降”，平均渲染时长下降。

---

## Phase 5：规模化渲染能力（P2）
目标：支撑“全量 + HQ + 多张”长期增长。

### 4.5.1 独立 GIF Render Worker 池
将 GIF 渲染从通用任务流程中抽象为独立能力池。

### 4.5.2 细粒度渲染任务表（可选）
建议新增：
- `archive.video_job_render_tasks`
- `archive.video_job_render_attempts`

用于：
- 失败重试细化
- 分布式调度
- 机器容量规划
- 产能监控

---

## 5. 里程碑与验收指标

## M1（1周内）
- 完成成本埋点 + 估算器升级。
- 可输出窗口级 `predicted_size/predicted_render_sec`。

验收指标：
- 估算偏差显著收敛（与当前版本对比）。

## M2（2~3周）
- 完成 bundling + mezzanine + bundle render 首版。
- 完成 weighted scheduler 首版。

验收指标：
- 不减张数前提下，job render stage P50/P90 明显下降。

## M3（3~5周）
- loop tune 缓存复用。
- HQ 子档策略上线。

验收指标：
- 主观 HQ 评分不下降；平均渲染耗时继续下降。

## M4（5周+）
- 独立 GIF 渲染 worker 池（视业务量决定）。

验收指标：
- 高峰期任务时长抖动显著降低，吞吐可线性扩展。

---

## 6. 明确不采用的“伪提速”方案
- 不通过“减少输出张数”换速度。
- 不通过“全局从 HQ 改成 light/standard”换速度。
- 不把 `gifsicle` 视为主提速手段（它是补充优化，不是主瓶颈解法）。
- 不把 GPU 当作 palette 链路的银弹。

---

## 7. 本版落地优先级清单（执行顺序）
1. 成本埋点 + 估算器重构（Phase 0）。
2. bundling + mezzanine + bundle render（Phase 1）。
3. weighted scheduler + 并发配置解耦（Phase 2）。
4. loop tune / 分析缓存复用（Phase 3）。
5. HQ 子档等质策略（Phase 4）。
6. 独立渲染 worker 池（Phase 5）。

---

## 8. 对产品目标的最终对齐
这条路线确保：
- 全量识别高光：保留。
- 不减张数：保留。
- HQ 输出定位：保留。
- 提速方式：从“减产降档”切换为“工程复用 + 调度优化 + 架构升级”。
