## 2. 先说结论：这一轮建议怎么改

我建议把这次开发拆成两层：

### 第一层：P0 最小可落地改造

目标：

- **只改 AI1 输入 payload**
- **不改 AI1 输出 schema**
- **不改数据库表结构**
- **不改 AI2/AI3 主逻辑**
- **尽量不影响现有审计接口**

这层改完，就能把 AI1 文本 JSON 从“工程调试包”升级成“任务合同”。

### 第二层：P1/P2 继续打通下游

目标：

- 让 AI1 的 task / constraint 真正影响 AI2 / worker / AI3
- 这部分后续做，先不和 P0 混在一版里

当前 AI1 持久化表：

- `internal/models/video_jobs.go:317`

这意味着：

但要注意：

## 5. 代码改造清单：按文件拆

---

### 5.1 `internal/videojobs/ai_gif_pipeline.go`

这是本次最核心的改造文件。

#### 当前关键位置

- AI1 主函数：
  - `requestAIGIFPromptDirective(...)`
  - `internal/videojobs/ai_gif_pipeline.go:637`

- 当前 payload 构造：
  - `buildDirectorPayload()`
  - `internal/videojobs/ai_gif_pipeline.go:781`

- 当前 user parts 构造：
  - `buildDirectorUserParts()`
  - `internal/videojobs/ai_gif_pipeline.go:810`

- 当前 usage metadata 记录：
  - `internal/videojobs/ai_gif_pipeline.go:898`

#### 建议改造动作

##### 动作 1：把 payload 构造从匿名 map 改成“可测试的 typed struct”

建议新增结构体，例如：

```go
type aiDirectorInputPayloadV2 struct {
    SchemaVersion   string                     `json:"schema_version"`
    Task            aiDirectorTaskContract     `json:"task"`
    Source          aiDirectorSourceContext    `json:"source"`
    NavigationHints *aiDirectorNavigationHints `json:"navigation_hints,omitempty"`
    RiskHints       []string                   `json:"risk_hints,omitempty"`
}
```

继续拆：

```go
type aiDirectorTaskContract struct { ... }
type aiDirectorSourceContext struct { ... }
type aiDirectorNavigationHints struct { ... }
type aiDirectorFrameRef struct { ... }
```

原因：

- 现在的 map 太容易混入日志字段
- typed struct 更容易单测
- 后续扩字段不容易失控

##### 动作 2：把 payload builder 从闭包中抽出来

当前 `buildDirectorPayload()` 是 `requestAIGIFPromptDirective()` 内部闭包。

建议改成独立 helper，例如：

```go
func buildAIDirectorInputPayloadV2(
    job models.VideoJob,
    meta videoProbeMeta,
    qualitySettings QualitySettings,
    inputMode string,
    frameSamples []aiDirectorFrameSample,
) aiDirectorInputPayloadV2
```

原因：

- 可单测
- 可复用
- 更方便未来加 schema 切换

##### 动作 3：保留 user parts 多模态装配逻辑，但让 text 部分只装 v2 payload

当前逻辑：

- `text` + `video_url`
- 或 `text` + `image_url[]`

建议：

- 这层结构不变
- 只替换 `text` 内容为 v2 payload

这样改造风险最小。

##### 动作 4：usage metadata 要继续保留工程诊断字段

当前 usage metadata 里有很多很有价值的工程诊断信息：

- `director_input_mode_requested`
- `director_input_mode_applied`
- `frame_count`
- `source_video_url_available`
- `source_video_url_error`
- `operator_instruction_version`
- `director_input_payload_v1`

建议：

- 这些字段继续保留到 metadata
- 但不要全部进入模型输入

推荐改成：

| metadata 字段 | 建议 |
|---|---|
| `director_input_schema_version` | 新增，值如 `ai1_input_v2` |
| `director_input_payload_v2` | 新增，记录最终发送给模型的 v2 payload |
| `director_input_mode_requested` | 保留 |
| `director_input_mode_applied` | 保留 |
| `frame_count` | 保留 |
| `source_video_url_available` | 保留 |
| `source_video_url_error` | 保留 |
| `operator_instruction_enabled` | 保留 |
| `operator_instruction_version` | 保留 |

关于 `director_input_payload_v1`：

- 如果当前没有外部消费者依赖这个 key，建议直接换成：
  - `director_input_payload_v2`
- 如果已有外部脚本依赖旧 key，建议 **灰度期双写一版**

##### 动作 5：persist context 直接写 v2，不改单表

当前持久化：

- `persist.InputContext`
- `InputContextJSON`

建议：

- 直接把 v2 payload 写进去
- 不做额外字段迁移

## 7. 提醒：Prompt 模板兼容性是这次改造最大的隐藏风险

这点很关键。

当前 AI1 的 system prompt 不是完全写死的，还会加载：

- fixed template
- editable template

代码：

- `internal/videojobs/ai_gif_pipeline.go:655`
- `internal/videojobs/ai_gif_pipeline.go:672`

这意味着：

> 如果你的线上 prompt 模板已经在用旧字段名，那么 payload 换成 v2 后，模板可能失配。

### 建议检查项

灰度前必须检查：

1. fixed prompt 里有没有引用旧字段语义
2. editable prompt 里有没有基于 flat payload 做说明
3. 是否存在运营侧约定：
   - `gif_target_size_kb`
   - `gif_profile`
   - `frame_manifest`
   - `source_video_url_available`

### 建议做法

P0 最稳妥方案：

- 同步更新 AI1 fixed/editable 模板描述
- 明确输入 schema 已从 flat metadata 升级为 task/source 结构

如果暂时不方便动模板，也要确认：

- 当前 prompt 没有硬依赖旧字段名

## 8. 灰度方案：推荐两种做法

### 8.1 简化版灰度（推荐）

适合：

- 当前 AI1 仅内部使用
- 外部依赖少

做法：

1. 直接切换到 v2 payload
2. usage metadata 新增：
   - `director_input_schema_version=ai1_input_v2`
3. 保留原工程诊断字段
4. 观测 1~3 天

优点：

- 改造最小
- 代码干净

### 8.2 稳妥版灰度（可选）

适合：

- 线上已有复杂 prompt 模板
- 你担心 schema 切换影响稳定性

做法：

增加一个**临时 schema 开关**，例如：

- `v1_flat`
- `v2_task_contract`

实现位置可选：

1. 临时环境变量
2. 临时质量配置
3. 临时代码常量 + 白名单 job

建议：

- 如果只是短期灰度，优先环境变量，不建议为了这次改造专门大动配置系统

## 9. 验证指标：怎么判断这次改造有没有价值

不能只看“功能没报错”，要看是否真的提升了链路质量与效率。

### 9.1 稳定性指标

重点观测：

- AI1 调用报错率
- AI1 JSON parse 失败率
- AI1 fallback 使用率

当前相关位置：

- `internal/videojobs/ai_gif_pipeline.go:942`
- `internal/videojobs/ai_gif_pipeline.go:953`
- `internal/videojobs/ai_gif_pipeline.go:963`

目标：

- 不高于改造前

### 9.2 token / payload 指标

重点观测：

- `director_input_payload_bytes`
- input token
- video token / image token

目标：

- 在 full_video 模式下，文本 payload 体积下降
- 无效字段减少

### 9.3 业务效果指标

重点观测：

- planner proposal 空结果率
- AI2 selected score
- `gif_candidates_v1.selected_count`
- AI3 deliver 率
- deliver fallback 率

目标：

- proposal 空结果率不上升
- deliver fallback 不恶化
- 候选质量至少持平

### 9.4 推荐验收标准

建议上线验收按以下标准：

1. AI1 fallback 率不升高
2. AI1 parse error 不升高
3. `director_input_payload_bytes` 明显下降或更稳定
4. AI2 proposal 空结果率不升高
5. 最终 deliver 率不下降

## 11. 本轮不建议一起做的事

为了控制风险，这一轮我不建议把下面几件事混进来：

1. 扩展 AI1 输出 schema
2. 让 AI2 同时改 prompt 与输出结构
3. 让 AI3 同时接 AI1 directive
4. 改本地 evaluation 权重
5. 引入 OCR/ASR/场景切分大模块
6. 扩视频元数据 DB 表

原因：

> 这会把“AI1 输入整改”变成“全链路重构”，上线风险太高，效果也难归因。
