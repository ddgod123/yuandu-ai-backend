# 数据模型与表设计细化

## 1. 设计原则

当前仓库已经有比较完整的数据落点，所以细化方案优先遵循：

1. **能复用现有表，就不轻易加新主表**
2. **新增实验信息优先落 metadata / feature_json**
3. **主链路表保留强结构，实验信息保留弱结构**
4. **所有 AI 决策要能串成完整链路**

---

## 3. 推荐的主链关系图

```text
archive.video_jobs
  -> archive.video_job_events
  -> archive.video_job_gif_candidates
      -> archive.video_job_gif_ai_proposals
          -> public.video_image_outputs (proposal_id)
              -> archive.video_job_gif_evaluations (output_id)
              -> archive.video_job_gif_ai_reviews (output_id/proposal_id)
              -> public.video_image_feedback (output_id)
  -> archive.video_job_gif_ai_directives
  -> ops.video_job_ai_usage
  -> ops.video_job_costs
```

这条链是你后续所有后台审计页面、回归分析、AB 对比的核心主链。

## 4. 各表细化建议

## 4.1 `archive.video_jobs`

### 当前作用

任务主表。

### 建议继续承载

- 主 stage / progress
- options 快照
- metrics 快照
- result collection id
- error message

### 建议在 `metrics` 中统一增加的键

- `pipeline_policy_version`
- `experiment_bucket`
- `gif_pipeline_sub_stage_status_v1`
- `delivery_decision_summary`
- `judge_hard_gate_summary`

### 不建议新增大量列

因为这张表已经承担主任务状态机，实验信息更适合落 `metrics`。

---

## 4.2 `archive.video_job_gif_candidates`

### 当前字段价值

- `start_ms/end_ms`
- `base_score`
- `confidence_score`
- `final_rank`
- `is_selected`
- `reject_reason`
- `feature_json`

### 建议强化 `feature_json`

建议统一沉淀这些字段：

- `reason`
- `scene_id`
- `scene_score`
- `motion_mean`
- `motion_peak`
- `clarity_estimate`
- `loop_estimate`
- `brightness_mean`
- `crop_applied`
- `source=local|planner_adjusted|rerank_adjusted`
- `diversity_group`
- `time_bucket`

### 作用

这样 candidate 将不仅能解释“为什么被选”，还能解释“为什么被淘汰”。

## 5. 新增表建议：尽量少、但可以有两个候选

## 5.1 可选新增：`ops.video_job_experiment_runs`

### 什么时候值得加

当你准备系统化做 A/B、灰度、分桶回归时。

### 建议字段

- experiment_id
- job_id
- bucket
- policy_version
- director_prompt_version
- planner_prompt_version
- judge_prompt_version
- candidate_policy_version
- created_at

### 什么时候不用急着加

如果当前所有实验都还能在 metadata 中管理，就先不建表。

---

## 5.2 可选新增：`ops.video_job_delivery_metrics_daily`

### 什么时候值得加

当你要做独立的经营指标看板，而 `baselines` 已经不够承载业务数据时。

### 建议统计

- 日期
- format
- deliver count
- download rate
- favorite rate
- share rate
- avg AI cost
- avg total cost
- top1 accept rate

### 当前阶段建议

先不建。优先从 `public.video_image_feedback + ops.video_job_costs + baselines` 聚合得到。

---

## 6. 字段命名与版本治理建议

凡是会影响线上效果的策略，都建议显式带版本：

- `prompt_version`
- `brief_version`
- `policy_version`
- `selection_version`
- `decision_policy_version`
- `experiment_bucket`

### 原则

只要一个值可能被线上调整，就必须可追踪。

---

## 7. 关系完整性要求

以下关系建议视为“强链路”：

1. `job_id`
2. `candidate_id`
3. `proposal_id`
4. `output_id`
5. `review/output`
6. `feedback/output`
7. `rerender -> proposal/output`

### 后台审计页必须能串出来

```text
job
  -> candidates
  -> proposals
  -> selected outputs
  -> evaluations
  -> ai reviews
  -> feedback
  -> rerender history
```

---

## 8. 最终建议

从数据设计角度，你当前系统已经够成熟。真正需要做的不是“大改表”，而是：

- 补 version 字段和 metadata 规范
- 补 experiment bucket
- 补决策链一致性
- 保证 `job -> candidate -> proposal -> output -> review -> feedback` 强可追溯

只要这条链足够稳，后续不管是改 prompt、改召回、改模型，都会非常好做。
