# 基于当前架构的视频转 GIF 最佳实践方案

> 目标不是推倒重来，而是在你现有架构上做“可上线、可回滚、可评估”的增量升级。

## 1. 先给结论：最佳实践架构应该是什么

对你这个系统来说，最佳实践不是“让一个大模型直接理解整段视频并输出 GIF”，而是：

> **多阶段候选流水线 + AI 语义增强 + 技术硬门槛 + 反馈闭环 + 人工兜底**

更具体一点：

```text
视频预处理
  -> 本地候选召回
  -> AI1 任务级策略
  -> AI2 候选语义提名/排序
  -> 规则层去重、控量、保多样性
  -> 渲染与结果评测
  -> AI3 成品语义复审
  -> 技术 hard gate
  -> deliver / internal / reject / manual review
  -> 反馈学习 / rerender / 基线回归
```

你当前已经实现了其中大部分骨架，因此接下来的最佳实践核心是**职责收敛与链路校准**。

---

## 2. 最核心原则：AI 只做“模糊判断”，不要做“确定性裁决”

### 2.1 适合交给 AI 的事

- 片段是否具备传播性
- 片段是否可脱离上下文成立
- 片段是否有情绪/反应/戏剧张力
- 哪个候选更像“值得交付的 GIF”
- 这个任务应该偏娱乐、偏设计素材、偏新闻摘录，还是偏社交传播

### 2.2 不适合交给 AI 的事

- 输入合法性校验
- 文件是否存在
- ffmpeg / ffprobe 是否可用
- 时间窗口是否合法
- 尺寸是否超标
- 文件大小是否超预算
- 渲染超时、上传失败、积分结算
- 技术评分 hard fail 的最终阻断

### 2.3 你的现有实现与最佳实践的契合度

当前代码已经大体符合这个原则：

- API / Worker / 渲染 / 存储 / 成本 由后端控制
- AI 主要介入 Director / Planner / Judge 三层
- Judge 之后还有技术 hard gate

所以方向不需要大改，重点是让三层 AI 的职责边界更清楚。

---

## 3. 最佳实践一：把 AI1 / AI2 / AI3 的职责边界再拉开

## 3.1 AI1 Director：只定义目标，不输出具体窗口

Director 最适合做的事情是：

- 定义任务目标
- 定义偏好权重
- 定义风险提醒
- 定义交付数量和时长区间

不建议让 Director 直接参与：

- 逐窗口打分
- 输出片段时间戳
- 对结果做逐条去留判断

### 为什么

Director 的价值在于**统一目标函数**，不是做细粒度执行。

如果 Director 过度参与窗口级决策，会和 Planner 职责重叠，造成：

- prompt 冗余
- token 浪费
- 结果不一致
- 难以归因

### 建议

保持你现在这种做法：

- Director 输出结构化 brief
- 由 Planner 消费 brief
- 所有字段尽量结构化、可落库、可回放

这个方向是对的。

---

## 3.2 AI2 Planner：只在候选集合上做“提名与排序”

Planner 是整条链路里最关键的 AI 阶段。

最佳实践上，Planner 不应该直接面对整段视频全量信息，而应该只面对：

- Top-K 本地候选窗口
- 每个候选的结构化特征
- 候选关键帧摘要（如果后续要加强）
- Director 的 brief
- 任务目标与约束

### 你当前已经做对的点

- 先本地召回候选，再让 AI 排序
- proposal 输出结构化 JSON
- 输出后继续做非重叠筛选与 threshold 控制

### 推荐再增强两件事

#### A. 候选多样性约束

当前非重叠约束很好，但还可以继续加入：

- 场景多样性
- 动作模式多样性
- 语义标签多样性
- 镜头位置多样性（前段/中段/后段）

否则 Planner 可能会连续选出同一情绪峰值附近的多个近似片段。

#### B. 候选摘要输入升级

如果后续要继续借助 AI 提高命中率，最有效的不是让模型直接看整段视频，而是给 Planner 增加每个候选的**关键帧包**或**摘要特征包**：

- 首帧/中帧/尾帧缩略图
- motion 强度
- face / emotion / OCR / subtitle 摘要（可选）
- 是否适合闭环的首尾相似度

这样 Planner 会比只看时间窗和简单 reason 更稳。

---

## 3.3 AI3 Judge：只做成品复审，不做硬性越权

Judge 的最佳实践定位应该是：

> **对已经生成出的 GIF 成品做“语义与可交付性”复审，而不是替代工程质量门禁。**

### 你当前做对的点

- Judge 看的是成品 output，不是原始候选
- Judge 输出的是建议，不是强制交付
- 后面还有 hard gate

### 推荐补充的两个规则

#### A. Judge 只能“降级”，不要“升级硬失败”

即：

- 可以把本来可交付的样本降成 `keep_internal` 或 `manual_review`
- 不能把技术 hard fail 的样本抬成 `deliver`

你当前 hard gate 其实已经在做这件事，建议把这个原则写成文档和代码注释中的显式规则。

#### B. Judge 应该看到“目标上下文”

Judge 不应只看技术评分，还应带上：

- Director brief 摘要
- Planner 的 proposal reason
- output 的技术评测
- 用户或场景类型

否则 Judge 会只看“像不像一个不错的 GIF”，而不是“像不像这次任务真正需要的 GIF”。

---

## 4. 最佳实践二：把本地候选召回做成真正的“第一生产力”

AI 上限取决于候选召回上限。

如果本地候选没有把最值得截取的窗口召回来，AI 再强也只能在错误集合中选相对好的答案。

### 4.1 你的当前优势

你已经有：

- local highlight scorer
- auto crop
- frame quality filter
- loop 相关指标
- feedback rerank

这已经比纯 prompt 式方案稳很多。

### 4.2 最佳实践建议

优先把候选召回做成三层：

#### 第一层：确定性规则召回

- scene cut
- motion 峰值
- 表情/反应峰值（如果后续补 CV）
- 字幕密度 / 对白冲击点（如果后续补 OCR/ASR）
- 亮度/清晰度/曝光基础过滤

#### 第二层：多样性控制

- 不同时间段
- 不同场景段
- 不同节奏类型
- 不同语义标签

#### 第三层：AI 语义重排

- Director 指令约束
- Planner 语义提名
- feedback rerank

### 4.3 推荐的候选规模

经验上可以把最终交付目标设为 `N`，而候选池至少保留 `3N ~ 8N` 的可比较候选。  
如果最终只交付 3 个 GIF，那么候选池尽量不要只有 3~4 个。

---

## 5. 最佳实践三：建立“统一目标函数”，避免 AI1/AI2/AI3 彼此打架

这是很多多阶段 AI 流水线最容易出问题的地方。

### 5.1 典型冲突

- Director 偏“传播性”
- Planner 偏“情绪峰值”
- Judge 偏“清晰稳定”
- 技术评分偏“效率与大小”

如果这些目标没有统一，最终就会出现：

- Planner 选得很激进
- Judge 频繁打回
- 技术 gate 再次打回
- rerender 和人工复审成本上升

### 5.2 最佳实践做法

建立一个统一的“交付价值函数”，至少包含四类权重：

- `semantic_value`
- `clarity_value`
- `loop_value`
- `efficiency_value`

Director 输出这些权重；Planner 与 Judge 都要消费同一套权重定义。

### 5.3 你当前已经接近这个结构

Director 已经输出：

- `quality_weights`
- `loop_preference`
- `duration_pref_min/max`

所以建议下一步不是加更多字段，而是：

- 保证 Planner / Judge 真正消费这些字段
- 在线上统计“Director 权重 -> 最终交付表现”的相关性

## 6. 最佳实践四：成本控制必须做成“分层触发”

如果每个任务都走满 AI1 + AI2 + AI3，全链路成本会不断上升。

### 6.1 推荐的成本控制策略

#### A. 按视频复杂度分层

- 短视频 + 本地高置信命中：可跳过 Director 或弱化 Planner
- 中等复杂视频：走 AI2
- 高价值 / 高复杂视频：走 AI1 + AI2 + AI3 全链路

#### B. 按用户价值分层

- 普通用户：默认轻链路
- 高付费用户 / 运营任务：全链路高质量模式

#### C. 按候选质量触发

如果本地 scorer 已经给出：

- 高置信度
- 高 loop 评分
- 高 clarity
- 高 output 预估

则可减少 AI 参与深度。

#### D. 按失败重试分层

- 首次任务：标准链路
- 二次 rerender：只重跑必要阶段
- 管理后台手工重渲：优先复用 proposal，不重做整条 AI 链

### 6.2 你当前已有的有利条件

你已经有：

- `ops.video_job_ai_usage`
- `ops.video_job_costs`
- rerender 能力
- prompt versioning

所以完全可以继续往“分层触发策略”升级，而不是统一重流程。

---

## 7. 最佳实践五：线上质量评估不要只看 AI 分，要看业务北极星

如果只看：

- AI judge 的 `semantic_verdict`
- evaluation 的 `overall_score`
- loop closure
- 文件大小

你会得到很多“技术上还不错”的 GIF，但未必是用户最喜欢的 GIF。

### 7.1 推荐把指标分成四层

#### 第一层：系统稳定性指标

- 任务成功率
- 阶段失败率
- AI fallback 率
- 平均处理时长
- 平均重试次数

#### 第二层：质量过程指标

- 候选召回数
- proposal 接受率
- Judge 降级率
- hard gate 拦截率
- rerender 率
- manual review 率

#### 第三层：结果质量指标

- deliver 率
- keep_internal 率
- reject 率
- top1 / topN 平均技术分
- loop / clarity / efficiency 均值

#### 第四层：业务效果指标

- 下载率
- 收藏率
- 使用率
- 分享率
- 用户二次编辑率
- 人工挑选与系统自动交付的一致率

### 7.2 当前代码里已有基础表

你已经有：

- `public.video_image_feedback`
- `ops.video_job_gif_baselines`
- `ops.video_job_gif_manual_scores`
- `ops.video_job_gif_rerank_logs`

这已经足够支撑一个较好的评估面板。

---

## 8. 最佳实践六：人工审核不要只是兜底，要变成训练数据工厂

很多系统把 `need_manual_review` 仅仅当成例外处理，但更好的做法是把它看作：

> **高价值标注入口**

### 8.1 哪些样本最值得人工标

优先抽样：

- Planner 高分但 Judge 打回的样本
- Judge 认为可交付但用户反馈差的样本
- hard gate 边界附近样本
- rerender 后翻盘样本
- 不同 prompt version 间结果差异大的样本

### 8.2 人工标完后要反哺哪里

- 调整 baselines
- 调整 Director 权重模板
- 调整 Planner prompt
- 校准 Judge recommendation
- 优化 feedback rerank 规则

### 8.3 你当前已经有很好的基础

因为现在 proposal / evaluation / review / feedback / manual scores 都有落点，所以人工审核已经不只是“人工看一下”，而是可以变成完整的训练闭环。

---

## 9. 最佳实践七：Prompt 治理要像“配置治理”，不是像“写文案”

你当前新增的 `ops.video_ai_prompt_templates` 非常关键。

### 9.1 推荐治理规则

每个 prompt 版本都至少带上：

- 版本号
- 生效范围（format / stage / layer）
- 是否启用
- 生效时间
- 变更说明
- 预期目标（提升什么）
- 回滚版本

### 9.2 推荐线上实验方式

- Director 版本 A/B
- Planner prompt A/B
- Judge prompt A/B
- 不同 prompt 对不同视频类型做分组实验

### 9.3 注意事项

不要同时大改：

- prompt
- 本地 scorer
- 评测阈值
- rerank 规则

否则效果变化将很难归因。

---

## 10. 最佳实践八：把重渲能力做成正式的运营工具链

你现在已经有按 proposal 重渲能力，这是非常好的基础。

### 10.1 重渲最佳实践场景

- AI Planner 认为值得，但首轮渲染参数不理想
- 管理员人工挑中某个 proposal，要求补出成品
- 新 prompt / 新质量策略上线后，对历史 proposal 做回放
- Judge 诊断建议明确指出“建议更短/更强调 loop/更紧裁切”

### 10.2 推荐后续增强

重渲时支持策略参数：

- 保持原 proposal，仅换渲染参数
- 保持原渲染参数，仅换窗口微调
- 对同一个 proposal 输出多个变体（传播型 / 轻量型 / loop 优先型）

这样重渲会从“修 bug 工具”进化成“运营实验工具”。

---

## 11. 最佳实践九：可解释性必须覆盖全链路

最好的 AI 流水线不是“最准的”，而是“出了问题能复盘的”。

### 11.1 一个任务至少要能回答这些问题

- 入口时为什么估算这笔积分？
- 本地为什么召回这些候选？
- Director 为什么给这组质量权重？
- Planner 为什么选这个 proposal 为第一名？
- 为什么这个 proposal 最后没交付？
- 是 Judge 打回，还是 hard gate 打回？
- 用户反馈后，为什么 rerank 发生了变化？

### 11.2 你当前已经具备的基础能力

- `video_job_events`
- `ai_directives`
- `ai_proposals`
- `ai_reviews`
- `gif_evaluations`
- `rerank_logs`

建议下一步把这些链成一个完整的 admin 审计视图：

```text
job -> candidate -> proposal -> output -> evaluation -> review -> feedback -> rerender
```

这会极大提高调参与线上排障效率。

---

## 12. 最佳实践十：建立分级回退策略，而不是单一 fallback

当前你已经有 AI fallback，这是对的，但还可以再细化成分级回退：

### 12.1 推荐回退分级

#### Level 1：Prompt/AI 失败

- Director 失败 -> 用默认 directive
- Planner 失败 -> 退回本地候选 topN
- Judge 失败 -> 输出技术评测 + 标记 manual_review 或默认内部保留策略

#### Level 2：视频质量不足

- 清晰度不足 -> 降级为 internal
- 时长不达标 -> reject
- loop 不理想 -> 标记 rerender 候选

#### Level 3：系统资源不足

- ffmpeg timeout -> 降级低质量模板
- 渲染失败 -> 缩短窗口重试
- 上传失败 -> 保留本地阶段快照，支持后续补偿

### 12.2 好处

这样每一种失败都能被“有业务含义地处理”，而不是简单记一条 error。

---

## 13. 推荐的目标架构图（文字版）

```text
[API CreateVideoJob]
  -> [源视频/格式/成本/积分校验]
  -> [Archive + Public Job Mirror]
  -> [Asynq Queue]
  -> [Processor]
       -> probe / crop / local highlight recall
       -> AI1 Director (brief)
       -> AI2 Planner (proposal ranking)
       -> feedback rerank / diversity / threshold
       -> candidate persist
       -> render + evaluate
       -> AI3 Judge (semantic review)
       -> hard gate
       -> upload / collection / emoji / package
       -> public sync / cost / settlement
       -> feedback / rerender / baseline / manual scores
```

这条链路里最重要的三个“不可妥协”点是：

1. **AI 永远不直接控制账务和状态机**
2. **hard gate 永远不被 AI 覆盖**
3. **所有关键决策都要可回放、可解释、可统计**

---

## 14. 最终建议

如果只给你一句最佳实践建议，我会说：

> **继续坚持“本地候选为底座，AI 做语义增强，结果靠技术门槛和反馈闭环校准”这条路线，不要退回到端到端黑盒选片。**

因为对你现在这个业务来说，真正重要的不是“AI 参与得更多”，而是：

- 更稳定
- 更可控
- 更便宜
- 更能持续优化

而你当前的代码基础，已经非常适合朝这个方向继续进化。
